## SPV
**SPV-01: 是否可以预防私自挖矿攻击**
- **描述**：spv其实就是为了保证链上区块提交到链下的合法性，保证链下提交的区块是真正在链上出块的。所以需要防止链下矿工节点作恶，提交一个链上不存在的区块来绕过spv相关验证。
- **检查要点**：检查如果矿工私自作恶是否spv可以预防此类攻击

**SPV-02: 是否检查了链上pow的相关信息**
- **描述**：如果没有针对pow检查链下提交区块的合法性，则可能导致spv的链下服务端停止工作，因为下一个正常区块由于接不上非法区块导致此验证作废。
- **检查要点**：检查spv验证是否验证了区块的header、chain work、pow等相关信息，得像一个正常链上节点验证矿工区块一样验证区块合法性。

**SPV-03: 存储的默克尔树验证是否完备**
- **描述**：如果链下存储的默克尔树验证检查不完备，则可以提交一个恶意的验证来绕过校验从而使得整个spv验证崩溃
- **检查要点**：检查默克尔树相关验证检查是否完备。

**SPV-04: Merkle 路径验证完整性**

- **描述**：确保 SPV 客户端在验证交易时，不仅检查 Merkle Root，还必须严格校验 Merkle Path 的哈希计算过程。

- **检查要点**：校验哈希函数（如 SHA-256）的实现是否与主链一致。检查是否存在对 Merkle 树深度或索引的边界检查，防止恶意构造的路径引发溢出或死循环。确保验证逻辑能够拒绝非法的、畸形的路径证明。

**SPV-05: 假根攻击防范 (Fake Root Attack)**

- **描述**：防止攻击者通过伪造一个带有合法签名的区块头（包含伪造的 Merkle Root），诱导 SPV 节点相信一笔不存在的交易。

- **检查要点**：审查 SPV 节点是否验证了区块头（Block Header）的 PoW（工作量证明） 难度。确保节点不仅验证交易在树中，还验证了该区块属于 最长合法链。检查是否集成了检查点（Checkpoints）机制以防止长距离回滚攻击。


**SPV-06: 日食攻击与布隆过滤器隐私 (Eclipse Attack & Bloom Filter Privacy)**

- **描述**：SPV 节点通常使用布隆过滤器（Bloom Filter）向全节点请求匹配的交易，若配置不当可能导致隐私泄露或被隔离。

- **检查要点**：审查布隆过滤器的 假阳性率（False Positive Rate） 设置，确保在隐私与带宽之间取得平衡。检查节点是否连接了多个互不关联的全节点，防止被单一恶意节点“日食”（提供虚假视图）。建议评估是否支持更安全的 BIP157/158 (Client-Side Filtering) 以替代传统布隆过滤器。

**SPV-07: 签名与业务逻辑关联性（防断章取义）**

- **描述**：审查签名结构体，确保包含所有影响业务逻辑的关键参数，防止签名被“断章取义”地使用。

- **检查要点**：确保签名涵盖了 Chain ID，防止在分叉链上进行重放攻击。验证签名数据中是否包含了交易的 Nonce 或 Sequence Number，防止重复执行。检查签名是否绑定了具体的合约地址或业务指令，确保签名不能在不同上下文（Context）中被复用。
