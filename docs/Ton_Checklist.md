# TON

- 1.是否严格校验 Jetton 资产流与回调消息？
  
    描述：Jetton 交互通常涉及 master、wallet 与通知回调多条消息。如果未校验消息来源、对应钱包关系、金额单位或回调 payload，可能被伪造通知导致错误记账或资产损失。
    
    建议：仅信任白名单中的 Jetton master 与派生 wallet，严格校验 transfer notification 的来源、金额与消息体结构，并为失败路径补充对账与回退处理。
    
- 2.是否存在错误的数据类型读写？
  
    描述：如果存储时写入的是 `uint`，读取时却按 `int` 解析，或者位宽定义前后不一致，就可能污染存储状态、破坏计算结果，甚至在后续消息中持续放大错误。
    
    建议：确保同一字段在写入与读取时使用完全一致的类型和位宽，并审查所有序列化与反序列化逻辑，避免出现类型错配。
    
- 3.是否为外部签名消息设置了过期时间？
  
    描述：如果签名消息没有过期时间，攻击者即使在很久之后获得旧签名，也仍可能尝试重放或复用该消息，扩大攻击窗口。
    
    建议：在外部消息中加入 `valid_until` 字段，并在当前时间超过该值时直接拒绝处理；将其与 `seqno` 或其他唯一标识一起使用。
    
- 4.是否遗漏了 `impure` 修饰符？
  
    描述：在 FunC 中，如果一个函数执行了改变合约数据（存储）、发送交易或接受消息等操作，但未声明为 `impure`，编译器可能把调用当作可省略表达式处理，导致关键逻辑未按预期执行。
    
    建议：审查所有具有改变合约数据的函数，确保它们显式标记为 `impure`；尤其检查负责更新状态、转账、接受消息和字典写入的函数。
    
- 5.是否在接收异步响应时校验了消息发送者身份？
  
    描述：TON 的跨合约取数和回调都依赖异步消息。如果在接收响应消息时不校验 `sender()` 是否为预期合约，攻击者可以伪造响应并污染业务逻辑。
    
    建议：所有回调、报价、通知和异步响应消息都应显式验证发送者地址，并在必要时同时校验 opcode、query_id 与关联上下文。
    
- 6.是否使用了不安全的随机数？
  
    描述：依赖 `random()` 等链上伪随机函数时，其结果与逻辑时间相关，攻击者可能通过枚举或观察当前区块环境来预测随机值，影响抽奖、分配等业务的公平性。
    
    建议：关键业务避免仅依赖链上随机数。优先采用“承诺-揭示”方案；若必须使用内置随机函数，可结合随机化逻辑时间降低预测难度，但不要在 external message receiver 中依赖随机化结果做高价值决策。
    
- 7.是否存在余额、份额或 LP Supply 更新不对称问题？
  
    描述：存入、提取、奖励结算等流程中，如果关键会计量只在部分分支更新，或只增不减、只写不读最新值，会导致账本偏移、奖励失真或用户权益被稀释。
    
    建议：将关键会计量更新收敛到统一逻辑中，为 deposit、withdraw、claim 等路径建立不变量测试，确保总量、份额和资产关系始终一致。
    
- 8.是否存在 Gas 预算不足导致部分执行或状态不一致的问题？
  
    描述：复杂业务流程在发起外部调用前如果未校验 `value` 是否足够覆盖 compute fee 和 forward fee，可能出现前半段状态已修改、后半段消息发送失败的情况。尤其是 out-of-gas 错误本身无法被合约内部捕获，最终容易造成状态不一致。
    
    建议：将 gas 预算作为接口前置条件，并尽量通过测试预估关键路径的 gas 消耗；在关键流程开始前校验费用是否充足，并为失败发送与 bounce 场景设计显式补偿逻辑。
    
- 9.是否存在跨消息流竞态条件导致的业务逻辑失效？
  
    描述：TON 的消息流可能跨越多个区块执行。如果在第一个消息中进行了权限或余额检查，但在后续异步回调（Callback）到达前，合约状态已被其他并发消息修改，则会导致逻辑失效。
    
    建议：避免假设状态在多条消息间保持不变。使用“Carry-value”模式（将必要状态随消息传递）或在处理后续消息时重新验证关键属性。
    
- 10.是否存在权限入口缺失或覆盖不全的问题？
  
    描述：合约存在升级、提币、参数修改、白名单调整或紧急暂停等敏感入口，但没有对 `sender`、签名或角色进行完整校验，可能被非授权消息绕过触发。
    
    建议：对所有敏感入口统一做权限校验，默认拒绝未知 opcode，并对升级、提币等高风险操作增加多签、双步骤确认或 timelock 机制。
    
- 11.是否错误使用了 SendRemainingBalance 或 SendRemainingValue？
  
    描述：`mode 128` 和 `mode 64` 会改变出站消息实际携带的价值。如果开发者误判其行为，可能意外发送整个账户余额或把入站剩余价值带入错误的消息。
    
    建议：仅在明确理解发送模式语义时使用 `SendRemainingBalance` 和 `SendRemainingValue`，并在关键转账路径中显式测试最终转出金额与剩余余额。
    
- 12.是否在异步回调失败时，合约是否具备完整的 Bounce 消息解析与逻辑回退机制？
  
    描述**：**在 TON 网络中，如果一个 `internal message` 设置了 `bounce = true`，当目标合约执行失败或目标地址不存在时，系统会自动触发一条回退消息。然而，**Bounced message 只会包含原始消息查询（Query）的前 256 bits**。[Non-bounceable messages | The Open Network](https://old-docs.ton.org/v3/documentation/smart-contracts/message-management/non-bounceable-messages)
    
    建议**：**确保触发回退逻辑所需的关键识别信息位于消息体的前 256 bits 内或者采用新版 Bounce 消息格式[Internal messages - TON Docs](https://docs.ton.org/foundations/messages/internal#new-bounce-format)
    
- 13.是否在验证前无条件接受外部消息？
  
    描述：在验证消息合法性（如签名、参数）前就调用 `accept_message()` 或 `SETGASLIMIT`，攻击者可发送大量无效消息，耗尽合约余额来支付Gas。
    
    建议：将 `accept_message()` 置于所有关键验证（如签名验证、参数检查）之后，确保只有合法消息才会消耗合约余额进行处理。
    
- 14.是否缺少外部消息的重放保护机制？
  
    描述：处理外部消息时未使用序列号（seqno）等机制，攻击者可重复发送已成功的消息，导致重复执行操作（如重复转账）。
    
    建议：在 `recv_external` 中，比较消息序列号与合约存储的序列号，验证通过后再调用 `accept_message()` 并更新存储的序列号。
    
- 15.是否将签名与交易上下文完整绑定以防抢跑？
  
    描述：如果签名消息没有绑定合约地址、接收地址、金额、`seqno`、`valid_until` 或 opcode 等关键上下文，攻击者可能复制签名抢先执行，甚至替换接收方后复用同一签名。
    
    建议：让签名至少覆盖 `to`、金额、`seqno`、过期时间和关键操作参数；同时实现重放保护，确保同一签名即使被公开看到也不能再次被利用。
    
- 16.是否校验了 `subwallet_id` 或 `wallet_id` 等实例标识？
  
    描述：对于包含实例标识的钱包或账户体系，如果不校验 `subwallet_id` 或 `wallet_id`，不同实例之间的签名消息可能被错误复用，带来账户混淆风险。
    
    建议：在外部消息验签后校验实例标识必须与合约当前配置一致，避免不同实例之间共享同一签名上下文。
    
- 17.是否在异步消息处理中重新验证关键状态以避免竞态？
  
    描述：由于TON消息处理是异步且可能跨区块的，在一个消息流执行过程中，合约状态可能被并发的另一个消息流改变，导致基于初始状态检查的逻辑失效（如双重支付）。
    
    建议：对于关键状态变更，采用严格的锁机制或检查-生效模式。不要仅依赖业务开始时的状态检查，在关键操作执行点应重新验证状态。
    
- 18.是否采用了 Carry-Value Pattern 设计跨合约资产流？
  
    描述：如果跨合约业务依赖“远端记录余额、我本地再记一份”这类分散式账本，而不是通过消息携带必要价值与状态上下文，就更容易在异步执行中出现竞态、双花或状态不一致问题。
    
    建议：在跨合约流程中尽量采用 carry-value pattern，让消息携带必要的价值和状态信息，减少对远端历史状态的隐式依赖，并在每一步重新验证关键条件。
    
- 19.是否在适用场景下优先使用了可 `Bounce` 的内部消息？
  
    描述：如果关键内部消息未启用 bounce，目标合约执行失败时发送方就无法收到失败反馈，进而难以及时恢复状态、退款或补偿，增加资产卡死和账本不一致风险。
    
    建议：对于需要失败感知和补偿的内部调用，优先使用 bounce-able message，并在发送方实现明确的失败处理与回退逻辑。
    
- 20.是否正确处理了地址格式与工作链校验？
  
    描述：未正确处理TON地址的可弹回（Bounceable）与不可弹回（Non-bounceable）格式，或未验证工作链（workchain），可能导致消息发送失败或发送到错误链。
    
    建议：使用标准库函数解析地址，并在发送消息前使用 `force_chain()` 等方法强制验证目标地址的工作链是否符合预期。
    
- 21.是否存在账户销毁的竞态条件？
  
    描述：使用 `send mode 128 + 32` 销毁账户时，如果缺乏前置校验，可能在处理其他未完成操作时意外销毁账户，导致资产丢失。
    
    建议：在销毁账户前，必须验证发送者权限，并确保合约没有待处理的操作或预期的后续消息。
    
- 22.是否充分测试了 hot upgrade 的迁移函数？
  
    描述：hot upgrade 依赖迁移函数在新代码上下文中即时执行。如果迁移函数读取旧结构错误、写入新结构错误或直接失败，合约可能立即进入不可用状态。
    
    建议：在 testnet 和本地沙盒中充分测试 `hotUpgradeData()` 一类迁移函数，覆盖旧结构、新结构和异常输入，确保失败不会让协议永久不可用。
    
- 23.是否存在整数溢出或下溢漏洞？
  
    描述：在进行投票权转移、代币转账等涉及整数运算的操作时，未对输入值进行边界检查，可能导致余额变为负数或溢出，破坏合约状态。
    
    建议：在所有数值运算前，使用 `throw_unless` 或 `require` 验证输入值的有效性（如 `from_votes >= amount`）。
    
- 24.是否存在有符号或无符号整数漏洞？
  
    描述：在进行投票权转移、代币转账等涉及整数运算的操作时，未对输入值进行边界检查，可能导致余额变为负数或溢出，破坏合约状态。
    
    建议：在所有数值运算前，使用 `throw_unless` 或 `require` 验证输入值的有效性（如 `from_votes >= amount`）。
    
- 25.是否校验了 Cell 边界与 Slice 解析安全？
  
    描述：TON 使用 cell、slice、builder 进行数据编码与解析。如果位宽、引用数量或解析顺序处理不当，可能触发 overflow、underflow 或越界读取，造成交易失败甚至被 DoS。
    
    建议：对 `storeUint`、`storeRef`、`load*` 等操作统一校验位宽和 refs 上限，对外部输入增加长度与深度限制，并用极限 payload 测试序列化边界。
    
- 26.是否正确使用了修改型与非修改型方法调用？
  
    描述：在 FunC 中，`.` 与 `~` 调用形式语义不同。若在字典删除、更新等操作中误用了非修改型方法，可能只拿到返回值却没有真正修改原始数据结构。
    
    建议：逐一检查字典和集合相关操作，确认需要原地修改的场景使用了正确的修改型调用形式，并通过测试验证状态确实被更新。
    
- 27.是否检查了关键函数的返回值？
  
    描述：如果忽略诸如字典删除、更新或其他关键操作返回的成功标记，合约可能在实际失败的情况下继续执行，最终造成逻辑错误和状态不一致。
    
    建议：对关键函数、字典操作和可能失败的辅助函数显式检查返回值，并在失败时使用 `throw_unless` 或 `require` 及时中止执行。
    
- 28.是否正确理解并覆盖了交易各个阶段的失败场景？
  
    描述：TON 交易包含 storage、credit、compute、action、bounce 等多个阶段。即使 compute 阶段成功，action 阶段仍可能因消息结构、发送模式或余额不足失败，如果审计时忽略阶段差异，就容易误判交易结果。
    
    建议：针对关键路径分别检查 compute 和 action 阶段的退出结果，并在测试中覆盖 bounce、发送失败、余额不足和结构错误等多阶段异常场景。
    
- 29.是否正确使用了预定义入口与 method_id？
  
    描述：TON 合约对 `recv_internal`、`recv_external`、`get` 方法等入口存在预定义约定。如果 method_id 设置错误，或把逻辑放在错误的入口中，可能导致方法不可达、消息被错误处理或安全校验失效。
    
    建议：检查所有入口函数是否使用了正确的预定义接口与 method_id，确保 internal、external 和 getter 逻辑边界清晰且不会互相混淆。
    
- 30.是否在外部消息中错误使用了不合适的发送标志？
  
    描述：对于外部消息，某些发送模式和标志并不适用。例如对外部消息使用 `+16` 没有实际接收 bounce 的发送方，容易造成错误的失败处理预期。
    
    建议：为出站外部消息只使用官方允许的模式组合，并避免在外部消息中使用不适合的 `+16`、`+32` 等标志。
    
- 31.是否校验了发送模式组合的合法性？
  
    描述：错误组合发送模式和标志，例如同时使用 `64` 与 `128`，会直接导致发送错误；如果业务逻辑没有覆盖这种情况，可能中断关键转账流程。
    
    建议：对所有发送模式做白名单约束，避免组合非法标志；对构造消息的公共函数加入模式校验和测试覆盖。
    
- 32.是否错误假设可以从其他合约同步拉取数据？
  
    描述：TON 合约之间不能像本地函数一样同步调用对方 getter。跨合约通信只能通过异步消息完成，如果业务逻辑错误依赖“立刻从另一个合约取回数据”，就容易出现设计错误与状态假设失效。
    
    建议：将跨合约取数设计为请求-响应式消息流，并在接收响应时校验消息发送者身份，不要在链上逻辑中假设可以同步读取另一份合约状态。
    
- 33.是否将敏感数据直接存储在链上？
  
    描述：TON 合约执行过程是透明的，运行期中的敏感值可以通过仿真或分析被恢复。如果把密码、私钥、密钥材料或其他秘密直接存入链上，即使做了简单包装，也不具备真正保密性。
    
    建议：不要在链上存储密码、私钥或其他敏感明文与密钥材料。链上仅保留必要的公开数据、承诺值或哈希，真正的敏感信息保留在链下。
    
- 34.是否执行了未经验证的第三方代码？
  
    描述：执行未受信任的第三方代码、库或动态加载代码会直接扩大攻击面，一旦代码内容被替换、伪造或本身包含恶意逻辑，合约安全性就可能被破坏。
    
    建议：在执行外部代码前校验代码来源、签名和安全属性；对新代码进行白名单或内容校验，避免直接执行未经验证的第三方代码。
    
- 35.是否使用了不规范的错误码？
  
    描述：如果合约直接使用 `throw(0)` 或 `throw(1)`，计算阶段会表现得像正常执行结束，从而让失败看起来像成功，增加排障、监控和审计的难度。不同模块复用同一错误码也会带来同样的问题。
    
    建议：建立统一的错误码分段规范，避免使用 0 和 1 作为异常退出码，并为高风险失败场景保留清晰、可追踪的错误编号。
    
- 36.是否妥善处理了 Gas 余量返还？
  
    描述：合约操作完成后，未将剩余的Gas价值返还给调用者，导致价值沉淀在合约中，用户体验不佳且资金效率低下。
    
    建议：在操作结束时，主动向消息发送者（`sender()`）发送一个带有 `Excesses` 操作码（`0xd53276db`）的响应消息，并设置 `SendRemainingValue` 模式以返还剩余价值。
    
- 37.是否覆盖了并发、失败与 Bounce 路径测试？
  
    描述：TON 合约的高风险问题往往出现在异步并发、消息失败、重复消息和 bounce 组合路径中。如果测试只覆盖主流程，很多真实漏洞会在上线后才暴露出来。
    
    建议：为成功、失败、重复消息、并发请求、低 gas 与 bounce 场景建立测试，并对关键交易的 exit code 与状态变化做断言。
    
- 38.是否缺少对外部代码的来源校验与执行控制？
  
    描述：执行未受信任的第三方代码、库或动态加载代码会直接扩大攻击面，一旦代码内容被替换、伪造或本身包含恶意逻辑，合约安全性就可能被破坏。
    
    建议：在执行外部代码前校验代码来源、签名和安全属性；对新代码进行白名单或内容校验，避免直接执行未经验证的第三方代码。
    
- 39.是否正确处理了交易阶段差异带来的异常与误判？
  
    描述：TON 交易包含 storage、credit、compute、action、bounce 等多个阶段。即使 compute 阶段成功，action 阶段仍可能因消息结构、发送模式或余额不足失败，如果审计时忽略阶段差异，就容易误判交易结果。
    
    建议：针对关键路径分别检查 compute 和 action 阶段的退出结果，并在测试中覆盖 bounce、发送失败、余额不足和结构错误等多阶段异常场景。
