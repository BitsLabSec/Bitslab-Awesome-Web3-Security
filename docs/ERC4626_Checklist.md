# ERC-4626 - Checklist

## ERC-4626
一、 标准合规性检查
- **ERC4626-01: 是否完整实现了 ERC-4626 必需接口？**
    - **描述**：部分ERC-4626实现虽然接口齐全，但 `convert*`、`preview*` 的返回值与真实存取款返回结果不一致，最终会导致报价与执行结果偏差。
    - **检查要点**：先核对全部函数签名与可见性，再验证三类一致性：`convert*` 与 `preview*` 口径一致、`preview*` 与实际执行结果可解释、`max*` 返回值与真实入口限制一致。

- **ERC4626-02: 只读查询函数在合理输入下是否稳定不回滚？**
    - **描述**：聚合器、前端、清算系统会高频调用 `asset`、`totalAssets`、`convert*`、`preview*`、`max*`。若这些函数在正常状态下回滚，会造成协议集成失败或可用性问题。
    - **检查要点**：覆盖空仓、满仓、极小额、极大额输入，确认函数行为稳定；异常仅限真正非法输入。

- **ERC4626-03: `preview*` 与实际执行结果是否一致？**
    - **描述**：`previewDeposit/previewMint/previewWithdraw/previewRedeem` 用于链下报价与风控，若与实际执行偏差过大，会引发用户损失或套利空间。
    - **检查要点**：同区块同状态下，对比 `preview*` 与真实执行返回值；确认不会给出可被利用的结果。

- **ERC4626-04: `max*` 函数是否正确？**
    - **描述**：`maxDeposit/maxMint/maxWithdraw/maxRedeem` 应反映协议限制与用户可操作数量的上限，而不是随意返回固定值或无意义值。
    - **检查要点**：验证限额、暂停、黑名单、策略容量、用户持仓变化下这些函数返回结果；无上述场景返回值应符合规范预期。

- **ERC4626-05: 存取款流程是否正确触发事件？**
    - **描述**：ERC-4626 要求存取款行为可被链下可靠追踪，事件缺失会导致索引、审计与监控系统失真。
    - **检查要点**：`deposit/mint` 触发 `Deposit`；`withdraw/redeem` 触发 `Withdraw`；份额变动同时发送 ERC-20 `Transfer` 事件。

- **ERC4626-06: 第三方代操作授权逻辑是否正确？**
    - **描述**：`withdraw`/`redeem` 支持由第三方代为执行时，必须严格校验份额授权，避免超额扣减或授权绕过。
    - **检查要点**：检查 owner != msg.sender 场景下 allowance 扣减是否正确；授权不足时应回滚。

- **ERC4626-07: 份额精度与资产精度处理是否一致？**
    - **描述**：vault share decimals 与 underlying asset decimals 可能不同；若缩放处理错误，会导致份额计算严重偏差。
    - **检查要点**：确认所有换算路径统一使用同一精度。

- **ERC4626-08: 零值与边界输入是否按预期处理？**
    - **描述**：零值输入、owner/receiver 边界地址、最小单位输入常是集成故障来源。
    - **检查要点**：明确并测试 `0` 金额行为；对 `receiver`、`owner`、`address(0)` 等边界地址做一致性校验。

二、 核心安全漏洞防护
- **ERC4626-09: 是否防范“空仓首存 + 捐赠”通胀攻击？**
    - **描述**：在总份额很小或为 0 的阶段，攻击者可通过首存与直接捐赠操纵资产/份额比率，稀释后续用户份额。
    - **检查要点**：引入初始虚拟流动性（virtual shares）或等效防护；校验首存路径不会被极小金额操纵。

- **ERC4626-10: 是否防范直接捐赠导致的汇率操纵？**
    - **描述**：攻击者可向 vault 直接转入资产（非 `deposit` 路径）改变 `totalAssets`，进而影响铸造/赎回价格。
    - **检查要点**：确认协议对捐赠行为有预期设计；检查捐赠后 `preview*` 与执行路径是否仍安全。

- **ERC4626-11: 舍入方向是否统一且不可循环套利？**
    - **描述**：ERC-4626 的多个换算路径涉及向上/向下取整，若方向不一致，可能形成“来回操作盈利”漏洞。
    - **检查要点**：交叉测试 `deposit/mint/withdraw/redeem` 与 `preview*`、`convert*`；验证不存在可重复获利的舍入循环。

- **ERC4626-12: 费用模型是否与会计一致？**
    - **描述**：费用问题的核心是“可对账且不双扣”。如果存款费、提款费、管理费、业绩费在 `preview`、实际执行、事件记录使用不同口径，或在资产侧与份额侧重复扣减，就会出现账实不符。
    - **检查要点**：按费用类型逐项核对四件事：计提时机、计提基数、扣费位置（资产侧/份额侧）、归集账户；再验证 `preview*`、执行结果与事件三者可一一对账。

- **ERC4626-13: `totalAssets` 是否完整反映真实管理资产？**
    - **描述**：资产可能分布在 vault、策略合约、待结算收益中。若 `totalAssets` 漏算或重复计算，会直接污染 share price。
    - **检查要点**：明确 `totalAssets` 口径（包含/不包含哪些资产）；检查跨合约资产、待领取收益、坏账处理逻辑。

- **ERC4626-14: 亏损场景下是否能正确反映净值下降？**
    - **描述**：vault 净值不一定只增不减。若在亏损时仍维持错误换算，会导致先赎回者套利、后赎回者受损。
    - **检查要点**：模拟策略亏损、坏账、罚没等场景，验证 share price 与赎回结果能反映真实损失。

- **ERC4626-15: 是否兼容 Fee-on-Transfer/通缩型资产？**
    - **描述**：若资产转账会扣手续费，`deposit(assets)` 的“声明金额”与“实际到账”不一致，易导致多铸份额。
    - **检查要点**：基于转账前后余额差计算实际到账资产；不要直接信任用户输入的 `assets`。

- **ERC4626-16: 是否兼容 rebasing / 生息型资产？**
    - **描述**：rebasing 资产余额会自动变化，不经过显式转账；若 vault 会计模型未考虑，可能引发分配错误。
    - **检查要点**：确认是否支持 rebasing；不支持时应显式限制或要求使用封装资产（如 wstETH 类资产）。

三、 外部交互与可操纵性
- **ERC4626-17: 是否防范闪电贷驱动的同区块操纵？**
    - **描述**：若 share price、费用、策略估值在单区块内可被快速拉动，攻击者可借闪电贷制造瞬时不公平换算。
    - **检查要点**：检查同区块多次存取款、策略交互、估值更新组合路径；必要时加入限速或时序防护。

- **ERC4626-18: 若 `totalAssets` 依赖外部价格，是否防范预言机操纵？**
    - **描述**：使用 DEX spot 或低流动性价格源估值底层资产，会导致份额价格被短时操纵。
    - **检查要点**：采用抗操纵价格源（TWAP、多源喂价、偏差阈值）；为估值异常设置熔断机制。

- **ERC4626-19: 外部调用顺序是否防重入？**
    - **描述**：资产代币可能带回调（如 ERC777 风格），策略调用也可能回调 vault；若状态更新顺序不当可被重入利用。
    - **检查要点**：遵循 CEI（Checks-Effects-Interactions）；关键入口添加 `ReentrancyGuard`；覆盖嵌套回调测试。

- **ERC4626-20: 策略交互失败或延迟赎回时，是否有可用性保障？**
    - **描述**：当策略赎回失败、队列拥堵、跨链回款延迟时，vault 可能出现提款阻塞或部分用户长期无法退出。
    - **检查要点**：设计提款队列、公平排序、部分成交与失败回滚策略；避免单笔失败阻塞全局。

- **ERC4626-21: 与非标准 ERC20 交互是否使用 `SafeERC20`？**
    - **描述**：部分代币不返回 bool 或返回行为异常，直接调用 `transfer/transferFrom` 易导致不可预期失败。
    - **检查要点**：统一使用 `OpenZeppelin SafeERC20`；对“无返回值/false 返回值”代币做兼容测试。

- **ERC4626-22: 是否考虑底层资产黑名单/暂停机制？**
    - **描述**：合规型代币可能随时拉黑地址或暂停转账，导致 vault 存取款功能突然失效。
    - **检查要点**：识别底层资产合规控制风险；提供失败处理与用户提示；必要时加入资产白名单策略。

四、 逻辑与权限管理
- **ERC4626-23: 敏感参数与管理操作是否有严格访问控制？**
    - **描述**：费用率、策略地址、资产上限、白名单、暂停开关等若可被未授权账号修改，会造成直接资金风险。
    - **检查要点**：所有管理入口使用明确角色控制（owner/governor/guardian）；最小权限原则落地。

- **ERC4626-24: 权限变更是否采用两步制与时间锁？**
    - **描述**：单步改 owner 或立即生效的关键参数变更容易因误操作或被盗私钥造成不可逆损害。
    - **检查要点**：采用 `transferOwnership + acceptOwnership`；关键治理动作增加 timelock 与链上公告窗口。

- **ERC4626-25: 可升级架构是否满足升级安全要求？**
    - **描述**：代理模式若存储布局错误、初始化函数保护不足或升级权限薄弱，会造成状态损坏或接管风险。
    - **检查要点**：核对 storage layout、initializer/reinitializer 防重复、upgrade 权限与回滚预案。

- **ERC4626-26: 资产上限/用户限额机制是否可绕过？**
    - **描述**：`maxDeposit/maxMint` 与真实入口校验不一致时，攻击者可通过边界条件或并发交易绕过限额。
    - **检查要点**：确保“查询上限”和“执行校验”使用同一逻辑；测试并发与抢跑场景。

- **ERC4626-27: `rescue/sweep` 函数是否可能误提底层资产？**
    - **描述**：紧急提币函数若允许提走 vault 管理资产，会直接破坏用户可赎回性。
    - **检查要点**：严格禁止 rescue underlying asset（或仅允许治理流程下受控执行并可审计）。

- **ERC4626-28: 暂停与紧急退出机制是否完整？**
    - **描述**：极端事件下，协议需要可控降级而不是“全停摆”或“带病运行”。
    - **检查要点**：明确暂停粒度（仅存入、仅铸造、仅策略投资等）；保留安全提款路径。

- **ERC4626-29: 是否存在循环或批处理导致的 Gas DoS 风险？**
    - **描述**：策略结算、收益分配、批量提款若依赖无界循环，可能在用户规模增长后永久不可执行。
    - **检查要点**：避免无上限循环；采用分批处理或拉取式领取；评估最坏路径 Gas。

五、 特殊场景下的检查项目:
- **ERC4626-30: Meta Vault 的 `totalAssets()` 是否只统计“本 vault 实际持有份额”对应资产？**
    - **描述**：多层 vault 场景里，`totalAssets()` 常见两类错误：漏算（未计入底层 yield）和重算（把不属于本 vault 的资产算进来），都会直接污染份额价格。
    - **检查要点**：用 `convertToAssets(balanceOf(address(this)))` 口径逐策略汇总；验证外部直接向策略转账不会抬高本 vault 的 share price。

- **ERC4626-31: 跨链份额迁移是否避免 `burn/mint` 导致的价值扭曲？**
    - **描述**：跨链桥接若在源链直接 burn share，可能短时改变本链 `totalSupply` 与汇率，造成在途用户被稀释或被套利。
    - **检查要点**：优先 lock/unlock 模式；检查跨链在途期间本地份额定价不被异常放大；验证失败回滚与重试不会重复记账。

- **ERC4626-32: Permit2 集成是否校验“记录金额”与“实际转入金额”一致？**
    - **描述**：若协议按 permit 金额入账，但 Permit2 实际按 requestedAmount 转入更小金额，会出现状态与真实资产不一致，可被利用提走超额资产。
    - **检查要点**：校验 `permitted.amount` 与 `requestedAmount` 关系；状态更新基于实际到账金额；拒绝“记录大额、实转小额”的情况出现。

- **ERC4626-33: Permit2 witness 哈希与 type string 是否严格符合 EIP-712 规范？**
    - **描述**：witness 哈希或类型字符串格式错误会导致签名验证行为与预期不一致，产生集成失败或安全边界缺失。
    - **检查要点**：witness hash 使用 `keccak256(abi.encode(typeHash, ...))`；type string 含 `witness)` 前缀与 `TokenPermissions` 等引用类型定义。

- **ERC4626-34: ERC1271/多签钱包场景是否防止签名碰撞与恶意合约签名滥用？**
    - **描述**：若业务 ID 只由 signature 推导，合约钱包（空签名 approved-hash）易出现 ID 冲突，或被恶意 ERC1271 合约用于低成本 DoS。
    - **检查要点**：ID 至少绑定 signer + nonce/requestId；校验 signer 与实际操作者关系；测试 Safe 等合约钱包的空签名路径。

- **ERC4626-35: `payable` 入口的余额门限判断是否排除 `msg.value` 膨胀效应？**
    - **描述**：`address(this).balance` 在函数执行时已包含 `msg.value`，若直接用于阈值判断可能被临时抬高/压低，从而绕过 gate。
    - **检查要点**：门限判断使用 `address(this).balance - msg.value`；核对后续 ETH 转发路径不会反向利用该差值。

- **ERC4626-36: `maxDeposit/maxMint/maxWithdraw/maxRedeem` 是否正确反映暂停与限额状态？**
    - **描述**：很多实现只在执行函数里做限制，但 `max*` 仍返回大值，导致聚合器/前端错误报价并触发失败交易。
    - **检查要点**：禁用存入时 `maxDeposit/maxMint == 0`；禁用取款时 `maxWithdraw/maxRedeem == 0`；与实际入口校验保持同源逻辑。

- **ERC4626-37: `deposit/mint/withdraw/redeem` 是否具备“零结果拒绝”与最小结果保护？**
    - **描述**：舍入到 0 的结果（0 shares 或 0 assets）会造成静默损失，是 donation/精度攻击常见入口。
    - **检查要点**：在核心路径拒绝零结果；路由层提供 `minShares/minAssets`；验证小额输入不会“扣资产但不给份额”。

- **ERC4626-38: 外部协议对接时，函数签名与参数路由是否逐项校验？**
    - **描述**：多协议集成常见“接口签名不匹配、owner/receiver 传错、approve 目标错误”，会导致提款失败或权限绕过。
    - **检查要点**：逐一核对外部 ABI；重点检查 `owner/receiver/spender` 语义与 approve 目标；对关键路径做端到端集成测试。

- **ERC4626-39: L2 价格依赖是否包含 Sequencer Uptime 防护？**
    - **描述**：在 L2 sequencer 异常期间，价格看似“新鲜”但不可用，仍继续估值会导致错误清算或错误铸赎。
    - **检查要点**：接入 sequencer uptime feed；downtime/恢复窗口内限制关键价格驱动操作；异常时降级到安全模式。

- **ERC4626-40: Permit 签名前跑是否有兼容策略（而非整体 DoS）？**
    - **描述**：permit 容易被前跑消耗 nonce，若协议未做兜底会导致正常交易反复失败。
    - **检查要点**：采用 try-permit + allowance fallback；失败时可继续走已有授权；避免把 permit 失败直接升级为全流程失败。

- **ERC4626-41: 管理员高影响操作是否具备防 MEV 约束？**
    - **描述**：调整费率、范围、路由、策略仓位等操作若可在极端价格瞬间执行，可能被夹子攻击利用，放大用户损失。
    - **检查要点**：为高影响操作加 timelock/冷静期/偏离阈值校验；执行前后验证关键安全指标（净值、抵押率、滑点）未异常。
