# DLMM/CLMM

**CLM-01 是否更新全局奖励费率？**

- **描述**：如果没有更新，可以在长时间未结算的时段末尾，抢先加入巨额流动性，迫使结算在变更后进行，从而把过去时段的奖励按`新总供给`分布，窃取旧 LP 的奖励。

- **建议**：在任何流动性更改的函数之前都先检查全局速率更新

**CLM-02 不按照比例添加流动性是否收取费用？**

- **描述**：用户在添加流动性的时候，如果不是按照池子比例添加就相当于一次swap。如果这里处理没有收取手续费，用户就可以通过添加流动性进行swap绕过交易费

- **建议**：添加流动性如果不是按照比例添加的时候需要加手续费

**CLM-03 未发放奖励是否正确处理？**

- **描述**：如果当前一个bin的流动性为0的时候，那么这一部分分到的奖励为0，但是奖励已经添加进去，这样会导致这笔资金丢失。

- **建议**：如果当前奖励没用就添加到下一个周期或者直接返回给一个地址。

**CLM-04 存在热土豆是否正确处理？**

- **描述**：在存在热土豆的模式下，热土豆是否有多个路经可以处理，如果存在多个路经，就可能存在绕过一些权限的认证

- **建议**：每一个热土豆都需要对应一个固定的操作类型。防止可以被多个路经处理。

**CLM-05 仓位变动的时候是否先结算然后更新费率？**

- **描述**：如果在添加流动性或者移除流动性的时候，需要先把之前用户对应的收益给结算来，如果不结算，用户在处理完之后结算用的流动性可能套出更多的资金。

- **建议**：在更新流动性的时候先结算手续费收益。

**CLM-06 精度损失的钱是否正确分配？**

- **描述**：如果在交易或添加流动性的时候，造成一定的精度损失，这一部分损失应该分给前面加入流动性的人，而不应该分配到后面进入的人，这样防止后进入的人直接套利。

- **建议**：在交易或者更新流动性的时候正确处理精度损失的部分。

**CLM-07 是否正确处理了仓位边界？**

- **描述**：如果在开仓的时候，记录的仓位上下限和实际开的仓位之间没做正确检查，就会在后续一些仓位的添加、移除等地方因为边界检查导致仓位无法处理，这样可能导致死仓风险。

- **建议**：开仓的时候正确检查仓位边界。

**CLM-08 是否正确处理了仓位边界？**

- **描述**：在做swap的时候，一般存在两种函数，给定输入计算输出或者给定输出计算输入。在处理这两种情况的时候需要保证优先倾向于项目方的利益。

- **建议**：给定输入计算输出向下取整，给定输出计算输入向上取整。

**CLM-09 是否正确处理滑点保护问题？**

- **描述**：dlmm因为每个bin的价格是固定的。所以单ticket之间不存在滑点问题。但是在多个ticket之间交换的时候就存在滑点保护问题。

- **建议**：添加滑点保护检查。

**CLM-10 是否对所有 ETH 转账或外部调用实施了重入防护？**

- **描述**：当合约执行 ETH 转账或调用外部代币合约时，若接收方为恶意合约，可能在回调过程中触发重入，在合约状态尚未完成更新前再次调用相关函数（如移除流动性），从而造成资产被重复提取的风险。

- **建议**：对所有涉及资产转移或外部调用的函数增加重入保护（例如使用 `ReentrancyGuard`）；同时遵循 **CEI**模式，在执行任何外部调用之前先完成必要的状态更新，以降低重入攻击风险。

**CLM-11 循环中是否存在外部调用或无界数组遍历？**

- **描述**：若在循环结构中执行外部调用，或遍历没有明确上限的用户/代币数组，当数组规模不断扩大时，交易所需的gas可能会逐渐增加，最终超过区块的gas限制，从而导致交易无法执行并产生**DoS**风险。

- **建议**：尽量避免在单笔交易中处理过大的数组；为相关数组设置合理的大小限制；同时考虑将批量发放逻辑改为由用户自行领取奖励，以降低单笔交易的 gas 消耗。

**CLM-12 是否对特权权限进行了合理限制，并通过多签或治理机制进行管理？**

- **描述**：若关键管理权限仅由单个私钥控制，一旦该私钥被盗、误用或持有者恶意操作，可能引发严重后果，例如资金被转移、协议参数被随意更改，甚至合约被暂停或锁定。这种设计也会形成明显的单点风险，影响系统的整体安全性和稳定性。

- **建议**：对管理权限进行细粒度划分，避免单一角色拥有过多控制权；对于敏感操作，建议通过多重签名钱包进行审批，以降低单点失误或密钥泄露带来的风险；同时为关键管理操作引入时间锁机制，为社区或用户提供必要的反应时间。

**CLM-13 是否有暂停状态检查？**

- **描述**：暂停机制的作用是在系统出现异常或安全问题时，能够迅速停止关键功能。如果该机制只在部分函数中实现，而某些关键操作（例如移除流动性、领取奖励等）未检查暂停状态，那么即使协议被暂停，这些函数仍然可以被调用，从而削弱暂停机制的效果。在漏洞尚未修复的情况下，这可能允许攻击继续发生或导致风险进一步扩大。

- **建议**：确保所有涉及状态变化或资金操作的关键函数都包含暂停状态检查。这样在触发暂停后，相关功能将被整体禁用，从而有效阻止进一步的交互并为问题处理提供缓冲时间。

**CLM-14 单个代币异常是否会影响整体领取流程？**

- **描述**：当协议支持发放多种奖励代币时，`getReward` 等函数通常会在循环中依次执行代币转账。如果其中某个奖励代币（例如 Token B）因转账被暂停、合约余额不足或被黑名单限制而导致转账失败，整个交易可能会回滚，从而使用户无法领取其他正常的奖励代币（例如 Token A）。这种设计会导致单个代币的问题影响到全部奖励的发放。

- **建议**：在处理多奖励代币的循环逻辑时，应避免单次失败导致整体回滚。若某个代币转账失败，可以记录对应的未领取金额并通过事件进行标记，使用户仍然能够成功领取其他奖励代币，从而避免单点故障影响整个奖励领取流程。

**CLM-15 在代理模式下，是否为未来升级预留了存储空间？**

- **描述**：在可升级合约架构中，如果父合约在后续版本中新增状态变量，而子合约没有提前预留额外的存储空间，可能会导致新的变量占用到原本已经使用的存储槽位，从而引发**存储布局冲突**，进而破坏合约状态或产生不可预期的行为。

- **建议**：在可升级合约的存储变量定义末尾预留一定数量的空存储槽。这样在未来升级时，可以安全地在父合约或当前合约中新增变量，而不会影响已有的存储布局。

**CLM-16 是否对添加奖励代币的权限进行了限制？**

- **描述**：如果添加奖励函数对所有人开放，攻击者可能添加行为异常的代币（例如在 `transfer` 时始终回滚、消耗大量 gas 或包含恶意逻辑的代币）。当奖励领取函数在循环中处理这些代币时，恶意代币可能导致整个交易失败，从而阻塞用户领取其他正常的奖励代币。

- **建议**：应限制 `addRewardToken` 的调用权限，仅允许 **Owner**、治理合约或 **DAO** 执行该操作；另外，在奖励领取逻辑中尽量避免单个代币的异常影响整体流程，例如对每个代币的领取过程进行隔离处理，以确保某个代币出现问题时不会阻止其他奖励的发放。

**CLM-17 是否存在抢先执行初始化函数？**

- **描述**：在代理模式下，逻辑合约通常不使用构造函数来初始化状态，而依赖 `initialize` 函数。如果该函数没有得到适当保护，攻击者可能在合约部署后、管理员初始化前抢先调用 `initialize`，将自己设置为合约拥有者，从而获得对合约的控制权。

- **建议**：在逻辑合约部署时立即调用 `_disableInitializers()` 禁用初始化函数，防止未授权调用；同时在部署脚本中将“部署逻辑合约 + 初始化代理合约”操作原子化执行，确保初始化只能由预期的管理员完成。

**CLM-18 原生代币分发是否采用了 自主领取模式？**

- **描述**：如果合约在发放奖励时直接调用 `address(user).transfer(amount)` 或类似方式，将资金主动发送给用户，当接收方为拒绝接收 ETH 的合约或存在高 gas 消耗的合约时，交易会失败。在批量分发场景下，单个恶意或异常用户就可能阻塞整个分发流程，影响所有用户的正常领取。

- **建议**：改用让用户主动调用领取函数，由用户自己提取奖励。这种方式可以避免单个用户阻塞整个发放流程，提高安全性和可用性。

**CLM-19 乘法运算是否进行了类型扩展？**

- **描述**：如果直接对 `uint64` 或 `uint128` 类型的数据进行乘法运算，结果可能超出原类型范围而溢出，导致交易失败或 Revert，从而引发 **DoS 风险**。

- **建议**：在计算前先将较小类型的数据提升到更大类型，再执行运算。

**CLM-20 闪电贷场景下，是否对参数和状态进行了安全处理？**

- **描述**：在合约允许闪电贷操作时，如果某些函数直接依赖 `balanceOf` 或实时链上余额进行计算或逻辑判断，攻击者可能利用闪电贷瞬间改变余额，从而进行**套利或操纵**，造成协议损失或收益异常。

- **建议**：引入全局状态变量或专门的参数来记录操作前的关键值，在闪电贷期间使用这些预先确定的参数而非实时余额进行计算，确保逻辑不被闪电贷操作干扰。

**CLM-21 CLMM跨ticket的时候流动性的更新是否正确？**

- **描述**：clmm中，流动性的增加减少是根据每个ticket点去更新的。如果在这个更新的时候出现问题，就会导致当前区间流动性更新错误。利用错误的流动性最后计算出来的输出代币数量就不对。

- **建议**：跨ticket更新流动性的时候严格检查流动性的变化情况。

**CLM-22 CLMM手续费计算是否正确？**

- **描述**： clmm当中手续费计算是间接的方式，首先计算ticket外的手续费，然后计算ticket内的手续费。如果在计算的过程中计算流程反了，就会导致用户计算出来的手续费不对。

- **建议**：计算手续费需要一步一步正确计算。先计算ticket内，在用全部减去ticket外。

**CLM-23 是否实现TWAP ？**

- **描述**：瞬时价格容易被攻击者通过大额交易、闪电贷或 JIT（Just-In-Time）策略短时间拉高或压低，从而在单笔交易或短时间内改变池内价格。依赖瞬时价格的逻辑（如清算、结算、奖励分配或跨合约定价）可能因此读取到被操纵的价格，导致**资金损失、不公平清算或错误奖励分配**。

- **建议**：采用**滑动窗口TWAP**，通过累积价格计算过去一定时间段的平均值，而非直接使用瞬时价格。根据不同应用场景调整 TWAP 窗口长度，以平衡抗操纵性与价格响应速度。

**CLM-24 是否存在 MEV 风险及前置/后置交易被利用的情况？**

- **描述**：交易可能被 MEV 搜索者或矿工/验证者通过重排序或插入交易（前置/后置）获利，从而损害普通用户或流动性提供者（LP）。典型风险包括用户遭受 **夹单（sandwich）攻击**，LP 的流动性被抽干、手续费或奖励被收割，以及协议声誉和资金安全受损。

- **建议**：对大额交易或可能跨越多个流动性区间的操作，设置更严格的滑点限制或分段执行，减少被操纵的空间。对短时间内频繁增减流动性的账户，引入冷却时间或操作限制，降低 MEV 攻击机会。

**CLM-25 关键参数变更是否具备最小延迟和预览机制**

- **描述**：如果管理员或治理可以即时修改关键参数，攻击者可能在参数生效的瞬间利用机会进行策略调整、抢占奖励或套利。缺乏提前预警或预估时，参数被恶意修改或私钥被滥用可能导致市场操纵、奖励分配异常，以及 LP 或用户资金损失。

- **建议**：对重大参数调整引入 **Timelock**，要求任何变更在经过固定延迟后才生效，让社区有时间监控和应对潜在风险。提供紧急应对机制，如“回滚”或 **emergency pause**，在参数被滥用或出现异常时能够快速停止敏感操作，降低潜在损失。

**CLM-26 Gas 消耗滥用** 

- **描述**：攻击者可能提交包含大量小额 bins、超长用户列表或异常参数的交易，使得批量操作函数（如奖励发放、跨-bin 状态更新、清算）在单笔交易中需要处理大量循环计算，从而耗尽 gas 导致交易失败或链上服务不可用。结果是合法用户无法顺利执行关键操作，攻击者借此降低系统可用性，形成 **DoS 攻击**。

- **建议**：限制单笔交易的操作复杂度，拒绝过大或异常的请求，从而降低通过高成本操作对系统造成干扰的风险。

**CLM-27 Tick没有流动性时，是否正确清除对应的费用和奖励速率？**

- **描述**：当最后一个用户移除流动性的时候，tick会为空，这个时候如果不清楚手续费速率和奖励速率，在第二次添加流动性的时候，就可能导致tick的速率是过期的，在后续计算的时候就会导致用户多分得手续费或者奖励。

- **建议**：tick为空时，清空当前奖励的手续费速率和奖励速率。
