# ERC-1155 安全审计清单

## 一、协议标准与元数据安全

- **ERC1155-1: URI 元数据占位符是否符合规范？**
  - **描述**：ERC-1155 要求 URI 必须支持 `{id}` 占位符以便客户端替换。
  - **建议**：确保 `uri()` 函数返回的字符串包含 `{id}` 字样，并符合 RFC 3986 规范。

- **ERC1155-2: URI 更新权限是否过于中心化？**
  - **描述**：如果管理员可以随意修改代币的 URI，可能会发生"图片掉包"或 Rug Pull 行为。
  - **建议**：对 `setURI()` 增加多签治理或时间锁（Timelock），或者在元数据确定后通过逻辑禁止进一步修改。

- **ERC1155-3: URI 的十六进制转换逻辑是否在链下处理？**
  - **描述**：根据 EIP-1155，链上 URI 通常只包含占位符，如果尝试在链上将 `uint256` ID 动态转换为 64 位十六进制字符串，会产生极高的 Gas 费。
  - **建议**：遵循标准，仅在链上返回包含 `{id}` 的静态字符串，将具体的 ID 转换逻辑留给前端或索引器。

- **ERC1155-4: URI 更新事件**
  - **描述**：是否在 URI 更新时发出 `URI` 事件。
  - **建议**：如需动态 URI，保留 `URI` 事件并在修改后 emit，客户端可通过事件重放同步。

- **ERC1155-5: 元数据 URI 的不可变性（Immutability）争议**
  - **描述**：某些项目方宣称 NFT 永久存储，但在合约中保留了 `setURI()` 权限且没有锁定机制，这属于中心化后门风险。
  - **建议**：提供一个 `lockURI()` 函数，一旦元数据确定且通过审计，允许管理员永久关闭修改权限，增加信任度。

- **ERC1155-6: URI 修改权限**
  - **描述**：是否存在任何人可修改 URI 或由单一管理员随意更改所有元数据。
  - **建议**：为 URI 修改定义独立角色（如 `URI_SETTER_ROLE`），并支持"锁定"机制：一旦锁定，不再允许修改 URI（或仅允许添加新 id 的 URI）。对关键 URI 记录哈希在链上，前端可校验元数据完整性。

- **ERC1155-7: 元数据格式规范**
  - **描述**：元数据是否符合 ERC‑1155 推荐的 JSON 结构（`name`, `description`, `image`, `attributes` 等）。IPFS/HTTP 链接是否稳定，有无单点依赖。
  - **建议**：使用 IPFS / Arweave 等去中心化存储，并在合约中记录内容哈希以防篡改。为重要资产提供链上 hash 校验字段，如 `bytes32 metadataHash[id]`。

- **ERC1155-8: ERC-165 接口支持是否正确实现？**
  - **描述**：其他协议依赖 `supportsInterface()` 来识别该合约是否符合 ERC-1155 标准，错误的实现会导致兼容性问题。
  - **建议**：确保重写 `supportsInterface()` 并正确包含了 `type(IERC1155).interfaceId`（即 0xd9b67a26）。

- **ERC1155-9: 批量余额查询（balanceOfBatch）是否越界？**
  - **描述**：查询多个账户及其对应 ID 的余额时，如果参数数组不匹配，会返回错误数据或报错。
  - **建议**：验证 accounts 数组与 ids 数组长度一致，并返回相同长度的 `uint256[]` 结果数组。

- **ERC1155-10: 批量转账事件（TransferBatch）是否正确触发？**
  - **描述**：前端索引器（如 Subgraph）依赖事件来追踪资产，错误的事件参数会导致数据展示错误。
  - **建议**：确保在 `_mintBatch()`, `_burnBatch()`, 和 `safeBatchTransferFrom()` 中准确释放 `TransferBatch` 事件，包含正确的 ID 和金额。

- **ERC1155-11: 市场兼容性**
  - **描述**：是否兼容主流市场的 ERC‑1155 要求（如正确回调、事件、URI）。
  - **建议**：使用主流钱包和市场（如 OpenSea 测试网）进行端到端测试，确认成功展示、挂单、购买、转移。如使用非标准扩展，在文档和接口中明确标注以免与通用工具不兼容。

## 二、权限控制与系统架构

- **ERC1155-12: 继承合约检查**
  - **描述**：是否继承：`ERC1155`、`ERC1155Supply`、`ERC1155Burnable`、`Pausable`、`AccessControl`/`Ownable` 等。
  - **建议**：优先使用 `ERC1155PresetMinterPauser` 等预设合约，在其基础上扩展，而非从零实现。

- **ERC1155-13: override 函数 super 调用**
  - **描述**：`override` 的 `_beforeTokenTransfer()` / `_afterTokenTransfer()` 是否 `super` 调用。
  - **建议**：如需自定义钩子，确保调用 super 并保留父合约的供应统计和暂停机制。

- **ERC1155-14: 敏感操作权限控制**
  - **描述**：铸造、销毁、暂停、URI 更改等是否仅限 owner 或角色调用。是否将关键权限放在单一 EOA 上。
  - **建议**：使用 `AccessControl` 定义 `MINTER_ROLE`, `PAUSER_ROLE`, `URI_SETTER_ROLE` 等。将 `DEFAULT_ADMIN_ROLE` 赋予多签钱包（如 Gnosis Safe）。对运维脚本加自动检查，部署后验证角色分配与预期一致。

- **ERC1155-15: 未授权敏感函数**
  - **描述**：是否出现未加限制的敏感函数（例如"测试函数"、"紧急函数"）。
  - **建议**：全文搜索诸如 `onlyOwner`/`onlyRole` 漏掉的函数，对所有修改状态、转移资产、升级逻辑的函数添加权限修饰器。将测试/调试函数完全移除或改为仅在本地/测试网络编译条件下存在。

- **ERC1155-16: 第三方转账的授权逻辑是否完备？**
  - **描述**：如果非代币持有者调用转账，必须检查该调用者是否获得了持有者的 `setApprovalForAll()` 授权。
  - **建议**：在转账函数中校验 `require(from == msg.sender || isApprovedForAll(from, msg.sender), "ERC1155: caller is not owner nor approved")`。

- **ERC1155-17: Approve-All 机制下的诱导授权风险？**
  - **描述**：由于 ERC-1155 只有全局授权（ApprovalForAll），用户一旦点击授权，所有 ID 的资产都会暴露给该操作员，这在钓鱼网站中极其危险。
  - **建议**：在 UI/UX 层面增加明确的风险提示，且在合约业务逻辑中，对于高价值资产，考虑增加二级权限验证（如二次签名确认）。

- **ERC1155-18: 铸造（Mint）权限是否受到严格限制？**
  - **描述**：若 `_mint()` 或 `_mintBatch()` 未受权限保护，任何攻击者都可以通过调用这些内部函数公开的包装器来随意增发代币。
  - **建议**：使用 `onlyOwner` 装饰器或 OpenZeppelin 的 `AccessControl` 模块来限制只有特定角色的地址可以执行铸造操作。

- **ERC1155-19: 销毁（Burn）权限是否存在越权风险？**
  - **描述**：销毁逻辑应仅限持有者本人或其授权的操作员调用，否则可能导致他人的资产被恶意销毁。
  - **建议**：在销毁逻辑中加入权限检查，确保 `require(from == msg.sender || isApprovedForAll(from, msg.sender))`。

- **ERC1155-20: 铸造权限与费用**
  - **描述**：是否允许任意用户调用铸造函数或以不合理低成本铸造。
  - **建议**：将铸造函数限制为 `onlyRole(MINTER_ROLE)`，如需用户铸造，采用受控的 `mint()` 入口（带支付、上限、白名单）。对铸造/销毁费用的计算使用 Solidity 0.8+，避免溢出；如有复杂数学，考虑库或额外检查。

- **ERC1155-21: 代理模式与初始化**
  - **描述**：是否采用代理模式（如 UUPS, Transparent Proxy），并正确初始化存储。
  - **建议**：使用 OpenZeppelin Upgrades 插件生成代理合约，并进行存储布局检查。限制升级权限为多签治理，并对每次升级进行独立审计与测试。

- **ERC1155-22: 可升级合约中的初始化函数防抢跑？**
  - **描述**：在代理模式（Proxy）下，合约部署后如果未立即调用初始化函数，攻击者可以抢先调用并设置自己为管理员。
  - **建议**：在构造函数中调用 `_disableInitializers()`，并确保部署和初始化在一个交易（Atomic Transaction）中完成。

- **ERC1155-23: 是否存在存储冲突（针对可升级合约）？**
  - **描述**：在使用代理模式时，如果基础 ERC-1155 合约的存储布局发生改变，会覆盖掉子合约定义的变量。
  - **建议**：在基础合约中预留 `uint256[50] __gap;` 以备未来扩展，并使用静态分析工具检查存储槽。

- **ERC1155-24: 存储 Gap 是否足以应对未来升级？**
  - **描述**：在复杂的 UUPS 升级模式中，如果父类合约（如 ERC-1155 基础库）升级增加了变量，而子合约没有足够的 __gap 空间，会发生变量覆盖。
  - **建议**：在所有可能被继承的合约末尾强制预留 `uint256[50] __gap;`。

- **ERC1155-25: 合约初始化函数（Initializer）是否可被重复调用？**
  - **描述**：对于使用代理模式的 ERC-1155，如果初始化函数未加保护，攻击者可以重置权限并接管合约。
  - **建议**：在初始化函数上使用 `initializer` 装饰器，确保该函数只能执行一次。

## 三、核心转账与资产逻辑

- **ERC1155-26: 自定义不安全转账函数**
  - **描述**：是否存在绕过 `safeTransferFrom()` 的自定义转移函数（例如直接修改余额映射、不检查 `onERC1155Received()`）。
  - **建议**：删除所有不走 `safeTransferFrom()` 路径的外部转账函数，除非仅对 EOA 且有明确注释与访问控制。如果必须支持"非安全转账"，明确限制为 `require(!to.isContract(), "no contracts");` 并在文档中提醒资产可能锁死的风险。

- **ERC1155-27: ERC-1155 接收者合约检查是否缺失？**
  - **描述**：在使用 `safeTransferFrom()` 时，如果接收地址是一个合约，必须验证其是否通过实现 `IERC1155Receiver` 接口来声明自己有能力处理该代币。
  - **建议**：确保在转账逻辑末尾显式调用 `_doSafeTransferAcceptanceCheck()`，并在目标合约返回非预期值时回滚交易。

- **ERC1155-28: 批量转账余额检查**
  - **描述**：在使用 `safeBatchTransferFrom()` 时，是否对每个 id 的余额/授权检查是否逐项执行，而非使用错误的聚合逻辑。
  - **建议**：使用 OpenZeppelin 原生 `_safeBatchTransferFrom()`，不要重写循环逻辑。如需扩展批量业务，在循环中每个 id 单独校验 `balanceOf()` 与权限，避免对数组长度及索引做复杂运算，保持 O(n) 线性简单逻辑。

- **ERC1155-29: 批量操作中的数组长度是否对齐？**
  - **描述**：在 `safeBatchTransferFrom()` 或 `balanceOfBatch()` 中，如果传入的 ids 数组和 amounts 数组长度不一致，会导致逻辑错位或索引越界。
  - **建议**：在函数开始处强制执行 `require(ids.length == amounts.length, "ERC1155: ids and amounts length mismatch")`。

- **ERC1155-30: 不可转让规则统一性**
  - **描述**：是否存在对特定 id 或账户不可转让的规则，但未在所有入口统一校验。
  - **建议**：将不可转让逻辑集中到 `_beforeTokenTransfer()` 中统一检查：对特定 id 或标记为 soulbound 的 token，禁止 from 与 to 同时为非零地址的转移。使用 `supportsInterface()` 暴露相应扩展接口，方便前端识别。

- **ERC1155-31: 自转账（Self-transfer）逻辑是否会损坏账目？**
  - **描述**：当 from 地址和 to 地址相同时，如果代码逻辑是先减去再加回，某些实现可能在中间状态报错或导致余额计算错误。
  - **建议**：编写单元测试覆盖自转账场景，确保在 `from == to` 时账户总余额保持不变。

- **ERC1155-32: 零地址转账检查是否缺失？**
  - **描述**：向 0x0 地址转账或从其铸造/销毁可能导致代币进入"黑洞"或破坏基础协议逻辑。
  - **建议**：在 Mint/Transfer 逻辑中强制执行 `require(to != address(0))`，在 Burn 逻辑中执行 `require(from != address(0))`。

- **ERC1155-33: 代币所有权转移时金融协议处理**
  - **描述**：在实施代币所有权转移时，需验证与该代币关联的所有有效金融协议（拍卖、贷款等）是否已被阻止或已妥善解决，然后再允许转移。
  - **建议**：建立金融协议状态检查机制，在转移前验证无活跃协议。

- **ERC1155-34: 供应上限约束**
  - **描述**：是否为各 id 设定最大供应或经济约束。`totalSupply(id)` 是否与实际铸造/销毁保持一致。
  - **建议**：继承 ERC1155Supply，使用其内建的 `totalSupply(id)` 维护逻辑。在 `_mint()` 前检查：`require(totalSupply(id) + amount <= maxSupply[id], "...");`，并在部署或初始化时配置各 `maxSupply[id]`。

- **ERC1155-35: NFT 唯一性约束是否在逻辑中体现？**
  - **描述**：如果将 ERC-1155 作为 NFT 使用（即发行量为 1），必须防止针对该 ID 的二次铸造。
  - **建议**：在针对 NFT 类型的 ID 进行 Mint 时，检查 `require(amount == 1 && totalSupply(id) == 0)`。

- **ERC1155-36: 所有权与使用权分离**
  - **描述**：是否区分"所有权 token"和"使用权/租赁状态"，避免协议逻辑让租客窃取所有权。回调中是否可以通过重入或异常状态永久占用 token。
  - **建议**：将所有权记录与使用权记录分离，例如：`ownerOf[id]`、`userOf[id]` + `expiresAt[id]`（使用权）。所有转账接口在转移所有权时自动清理/更新使用权状态。对涉及租赁与回调的函数使用 nonReentrant，并在回调前锁定关键状态。

- **ERC1155-37: 批量转账的 Gas 极限与 DoS 风险？**
  - **描述**：批量操作涉及循环，如果传入的数组过长，交易消耗的 Gas 可能会超过区块上限，导致功能失效。
  - **建议**：为批量操作设置合理的上限（如 `MAX_BATCH_SIZE`），防止无限循环导致的拒绝服务（DoS）攻击。

- **ERC1155-38: 资产锁死风险**
  - **描述**：是否可能将 token 锁在不支持 ERC‑1155 的合约中。
  - **建议**：通过 `safeTransferFrom()` 强制检查接收合约是否实现 `IERC1155Receiver` 接口，避免无回调合约锁死资产。如业务必须支持向任意合约发送 token，考虑添加受限的"紧急恢复"函数，由多签在严格条件下取回误锁资产。

- **ERC1155-39: 暂停机制（Pausable）是否会导致资产锁死？**
  - **描述**：如果合约包含 `pause()` 功能，一旦管理员私钥丢失或恶意暂停，所有用户的资产将无法转账、铸造或销毁。
  - **建议**：明确定义暂停的触发条件，并考虑设置"最长暂停期限"或"仅限紧急情况下暂停提取"的逻辑。

## 四、外部交互与重入防护

- **ERC1155-40: safeTransferFrom 后外部调用**
  - **描述**：在 `safeTransferFrom()` 后是否调用其他外部合约/复杂逻辑。
  - **建议**：将所有状态更新放在调用 `safeTransferFrom()` 之前（检查‑效应‑交互模式）。对可能被回调重入的函数增加 `nonReentrant`（继承 `ReentrancyGuard`）。

- **ERC1155-41: 业务流程重入风险**
  - **描述**：业务流程（租赁、抵押、订单撮合）是否在回调中可被重入。
  - **建议**：尽量不要在回调函数中执行其它外部调用，如需执行，拆分为用户主动调用的独立函数。

- **ERC1155-42: 是否存在由于接收者钩子引发的重入风险？**
  - **描述**：转账函数会触发外部合约的 `onERC1155Received()` 钩子，攻击者可能利用此回调在余额更新完成前再次进入合约执行操作。
  - **建议**：严格遵循"检查-生效-交互"（CEI）模式，在触发外部回调之前完成所有内部余额状态的更新，或使用 `nonReentrant` 装饰器。

- **ERC1155-43: 铸造（Mint）与销毁（Burn）中的钩子重入风险？**
  - **描述**：不仅是转账，部分 ERC-1155 实现在 `_mint()` 或 `_burn()` 逻辑中也会触发 `onERC1155Received()`，如果这些操作后还有未完成的状态更新，同样会引发重入。
  - **建议**：在执行 `_mint()` 或 `_burn()` 操作时，确保余额更新是该函数的最后一个状态改变操作，或者统一使用防重入锁。

- **ERC1155-44: 接收回调安全校验**
  - **描述**：接收合约是否在 `onERC1155Received()` / `onERC1155BatchReceived()` 中正确校验传入的 `operator`, `from`, `ids`, `values`, `data`。是否对任意 ERC‑1155 token 赋予了过大的信任（例如只要转入就自动获得权限）。
  - **建议**：在接收回调中验证：`msg.sender` 必须是预期的 ERC‑1155 合约地址（白名单）。ids 是否在允许范围，values 是否合理。对赋权逻辑增加白名单/授权列表，仅对特定 token id 或来源合约生效。

- **ERC1155-45: 转账逻辑中存在对 msg.data 的非预期解析？**
  - **描述**：`safeTransferFrom()` 最后一个参数是 `bytes data`，如果合约尝试解析这段数据来执行某些操作（如自动质押），可能被构造恶意指令。
  - **建议**：对 data 参数的解析必须经过严格的白名单校验，或者仅将其透传给接收者钩子，严禁在当前合约中根据此数据执行高危权限操作。

- **ERC1155-46: 恶意接收者 DoS 攻击风险？**
  - **描述**：恶意合约可以在 `onERC1155Received()` 中直接 `revert()`，从而阻止任何向其转账的行为，这可能导致拍卖等业务流程卡死。
  - **建议**：在设计涉及"强制转账"的业务逻辑时，优先采用"拉取（Pull）"模式而非"推送（Push）"模式。

## 五、数学计算与业务边界

- **ERC1155-47: 溢出与抢跑保护**
  - **描述**：是否使用 Solidity 0.8+，避免手动实现 SafeMath。是否存在价格、手续费、奖励计算，可能被前置交易利用（抢跑）。
  - **建议**：统一使用 Solidity 0.8+ 编译版本，关掉自写 SafeMath。对拍卖、竞价等敏感逻辑采用 commit‑reveal 或私有交易（如 Flashbots）模式。

- **ERC1155-48: 余额更新是否防止溢出（针对旧版本 Solidity）？**
  - **描述**：在 Solidity 0.8.0 之前，减法和加法可能导致整数溢出。
  - **建议**：如果是旧版本合约，必须使用 SafeMath；0.8.0+ 版本则需关注是否在 unchecked 块内执行了不安全的加减。

- **ERC1155-49: 自定义逻辑中的余额双花（Double Spend）漏洞？**
  - **描述**：如果在继承 ERC-1155 的基础上增加了"余额质押"或"锁定"功能，若逻辑编写不当，可能导致锁定的余额仍能通过标准的 `safeTransferFrom()` 转出。
  - **建议**：重写 `_beforeTokenTransfer()` 钩子，在所有转账发生前校验当前 ID 的可用余额（Total - Locked），确保转账金额不超过可用部分。

- **ERC1155-50: 代币 ID 是否存在"幽灵资产"注入？**
  - **描述**：如果业务逻辑中包含"ID 预注册"，而转账逻辑未校验该 ID 是否已正式激活，攻击者可能通过 `_mint()` 一个未定义的 ID 来操纵某些依赖"总 ID 数量"的统计逻辑。
  - **建议**：建立一个内部映射表记录已激活的 ID，并在 Mint 或 Transfer 时进行 isValidID 校验。

- **ERC1155-51: 集成 DeFi 协议时的闪电贷价格操纵？**
  - **描述**：如果该 ERC-1155 代币被用作借贷抵押品，且其价格计算依赖于 Uniswap 等流动性池，攻击者可通过闪电贷瞬间改变 ID 的价值。
  - **建议**：使用 TWAP（时间加权平均价格）预言机，严禁使用瞬时池余额（Spot Price）作为资产定价依据。
