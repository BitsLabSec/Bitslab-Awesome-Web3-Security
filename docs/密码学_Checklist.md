
### 密码学安全 (Cryptography) 审计清单

#### 一、 签名验证安全

**CRYPTO-01: 是否防范了签名延展性（Signature Malleability）攻击？**
- **描述**：在 secp256k1 曲线中，每个合法签名都有一个对称的合法签名。如果合约未限制 `s` 的取值范围，攻击者可以篡改签名但仍通过验证，导致双花或重放。
- **检查要点**：确保使用 OpenZeppelin 的 `ECDSA` 库进行验证，该库会自动检查并拒绝高 `s` 值的签名。或者使用的官方的密码学仓库，密码学相关的审计者应该查看对应依赖库是否包含此类检查


**CRYPTO-02: 签名验证是否包含防止重放（Replay）的 Nonce 机制？**
- **描述**：缺乏唯一随机数（Nonce）的签名可以被截获并多次重复调用，导致用户资产被多次扣除或权限被重复触发。
- **检查要点**：签名哈希中必须包含 `nonce`，且合约在验证成功后应立即将该 `nonce` 标记为已使用（如 `nonces[user]++`）。


**CRYPTO-03: 签名验证是否包含防止重放（Replay）的绕过nonce的机制**
- **描述**：即便存在nonce检查，是否可以通过绕过nonce校验来达成目的。
- **检查要点**：比如有的代码当中，由于pending当中的消息都会加入到内存池当中，此时高于当前储存的nonce的签名也可以通过校验这就违背了交易的顺序性原则，总之就是检查nonce的严格完整和逻辑的契合性。


**CRYPTO-04: 签名是否包含 Domain Separator（EIP-712）以防止跨协议重放？**
- **描述**：如果签名不包含合约地址或链 ID，攻击者可以将用户在 A 协议的签名拿到 B 协议使用，或在测试网的签名拿到主网使用。
- **检查要点**：检查是否实现了 EIP-712 标准，签名哈希中应包含 `DOMAIN_SEPARATOR`（包含 `verifyingContract` 和 `chainId`）。


**CRYPTO-04: 是否检查了 ecrecover 返回零地址的情况？**
- **描述**：`ecrecover` 在验证失败时返回 `address(0)`。如果合约逻辑未检查此返回值，且权限变量（如 `owner`）恰好为零地址，攻击者可绕过验证。
- **检查要点**：验证逻辑中必须包含 `require(recoveredAddress != address(0), "Invalid signature")`。


**CRYPTO-05: 是否检查了BLS聚合签名算法防止“Rogue Key Attack”的情况？**
- **描述**：bls等类似签名算法可以被Rogue Key Attack，简单来说此类聚合签名算法聚合的签名是通过双线性映射的加法聚合成了一个新的聚合公私钥进行签名，如果攻击者通过知道了聚合公钥和自己的公私钥，就可以自己伪造聚合签名，详细要点可以搜索bls-Rogue Key Attack。
- **检查要点**：解决方法是要求签名者提供“公钥知识证明”（Proof of Possession）。如果无法提供类似聚合签名算法则是不安全的。


**CRYPTO-06: 是否检查了DSA类签名算法中随机数K的随机性问题？**
- **描述**：DSA类签名算法如果两次签名使用了相同的随机数K或者二者有依赖关系关联性，那么签名的私钥就可以被解出来。
- **检查要点**：签名算法是否是官方实现，或者随机数K是否适用的安全的随机数。


**CRYPTO-07: 签名的消息是否包含了所有应该包含的内容？**
- **描述**：签名算法的消息如果没有包含全部应该包含的内容那么未被签名的消息内容部分攻击者可以篡改。
- **检查要点**：签名算法签名内容应当包含所有应当包含的内容，具体根据业务实际情况而定


**CRYPTO-08: 验证签名的算法接口是否包含签名的公钥等信息？**
- **描述**：签名算法如果接口可以传公钥，那么攻击者可以自己生成一个公钥和对应签名绕过此签名检查
- **检查要点**：签名的公钥应当由验证方指定，而不是用户指定


**CRYPTO-09: 验证签名的算法的私钥是否明文存储在配置文件当中？**
- **描述**：如果私钥写在配置文件当中那么会造成私钥泄露，整个加密体系崩溃
- **检查要点**：私钥不可以明文存储。

**CRYPTO-10: 是否使用了过时或不安全的哈希算法？**
- **描述**：签名过程中使用的哈希算法（如 MD5 或 SHA-1）如果存在已知的碰撞风险，攻击者可以构造两个不同的消息产生相同的签名，从而伪造授权。
- **检查要点**：确保在签名哈希过程中使用 `keccak256` 或 `SHA-256` 等强抗碰撞哈希算法。

**CRYPTO-11: 是否对签名消息进行了正确的哈希处理？**
- **描述**：如果直接对原始长消息进行签名而不先进行哈希，或者哈希逻辑在链上链下不一致，会导致验证失败或被攻击者利用编码漏洞。
- **检查要点**：验证签名逻辑是否遵循 `sign(keccak256(message))` 的标准流程，并确保链上验证时重建的哈希与链下一致。

**CRYPTO-12: 是否正确处理了多签（Multisig）中的签名排序？**
- **描述**：在需要多个签名的场景下，如果合约未要求签名按地址顺序排列，攻击者可能通过调换签名顺序来规避重复签名检查。
- **检查要点**：在多签验证循环中，确保检查 `currentOwner > previousOwner`，强制要求签名按地址升序排列。


#### 二、 哈希函数与数据完整性

**CRYPTO-13: 在使用 keccak256 时是否防范了哈希碰撞（Hash Collision）？**
- **描述**：使用 `abi.encodePacked()` 处理多个动态类型参数时，不同输入可能产生相同结果（如 `("a", "bc")` 与 `("ab", "c")`）。
- **检查要点**：在对多个动态参数进行哈希前，应使用 `abi.encode()` 代替 `abi.encodePacked()` 以确保编码的唯一性。

**CRYPTO-14: 关键业务逻辑是否依赖了可预测的“随机数”？**
- **描述**：使用 `block.timestamp` 或 `blockhash` 生成的随机数可被矿工预测或操纵，不适用于高价值业务。
- **检查要点**：严禁使用链上参数作为核心随机数源，应采用 Chainlink VRF 等可验证的链下随机数方案。

#### 三、 默克尔树验证 (Merkle Proof)

**CRYPTO-15: 默克尔树验证是否防范了“叶子节点注入”攻击？**
- **描述**：如果叶子节点长度与中间节点相同，攻击者可能将中间哈希值伪造成叶子节点提交，从而绕过白名单。
- **检查要点**：在计算叶子哈希时进行二次哈希，或在哈希前添加特定前缀（如 `0x00`）以区分层级。

**CRYPTO-16: 默克尔树根（Root）的更新是否有权限控制？**
- **描述**：如果更新 Root 的函数缺乏访问控制，攻击者可以替换为恶意 Root，从而控制所有验证路径。
- **检查要点**：确保 `updateMerkleRoot` 等敏感函数仅限 `onlyOwner` 或多签钱包调用。

**CRYPTO-17: 默克尔树的计算方式是否是标准形式？**
- **描述**：默克尔树的计算方式是中间节点的结果是两个子阶段的拼接的hash结果，如果包含顺序或者相关省略信息，那么默克尔树的根节点的hash值可能有多个子树对应，这是不正确的。
- **检查要点**：确保一个默克尔树只有一种构造方式，树的结构不能发生改变。

**CRYPTO-18: 默克尔树证明方式是否可以通过传入空子树来绕过校验？**
- **描述**：默克尔树的计算方式如果可以传入空子树直接传入树根来绕过验证则摧毁了默克尔树的整个验证逻辑。
- **检查要点**：确保一个默克尔树不可以传入空子树。

**CRYPTO-19: 默克尔树证明传入的子树需要包含自身存储相关信息，当心恶意的攻击者使用其他人的证明来伪造身份认证**
- **描述**：默克尔树的计算方式如果证明可以被重放或者伪造，则可以破坏整个默克尔树的验证逻辑。
- **检查要点**：确保一个默克尔树的证明不可以伪造或者重放。



#### 四、 逻辑与权限管理

**CRYPTO-20: 链上敏感数据是否误用了“私有变量”进行加密？**
- **描述**：`private` 变量仅限制合约访问，其数值在链上完全公开。存储明文密码或私钥片段在其中不具备保密性。
- **检查要点**：审查是否存在试图通过 `private` 关键字保护敏感数据的逻辑，敏感信息应仅在链下处理。

**CRYPTO-21: 预签名（Off-chain Signing）是否包含过期时间（Deadline）？**
- **描述**：没有过期时间的签名永久有效。如果市场环境变化或用户意愿改变，旧签名仍可能被攻击者利用。
- **检查要点**：签名结构体中应包含 `deadline`，合约验证时需检查 `require(block.timestamp <= deadline, "Signature expired")`。

**CRYPTO-22: 是否考虑了“签名撤销”机制？**
- **描述**：用户可能需要作废已发出但未使用的签名（如取消订单）。
- **检查要点**：检查合约是否提供撤销功能，通过手动标记 `nonce` 已使用或维护一个 `revokedSignatures` 映射来实现。

#### 五、 加密算法检查要点

**CRYPTO-23: AES加密算法如果使用了CBC模式，是否可以预防padding oracle attack？**
- **描述**：AES-CBC模式是一种类常见的加密算法，项目使用的加密算法是否考虑了padding oracle attack这类攻击方式。
- **检查要点**：具体检查要点需要详细查询padding oracle attack了解其中原理，像是unisat的密码学漏洞都是根据这个漏洞思路挖掘并发现的。核心是解密的时候很多库函数解密是否成功的判定条件是AES-CBC的解密函数是否报错，而不是和加密原文比较，这就存在这类问题，是否报错并不能和是否解密成功划等号，也就是说有多重可能对应此类结果，存在爆破或者各种破坏协议的风险。

**CRYPTO-24: 非对称加密算法是否适用rsa等非标准算法？**
- **描述**：非当前安全性的标准算法会造成潜在的资产损失。
- **检查要点**：检查使用的加密算法是否为非标准算法。

**CRYPTO-25: 随机数生成算法是否无法预测？**
- **描述**：由于链上随机数一直是一个问题，所以这个问题每次审计相关内容需要集中思考这个问题，随机数是否可以预测，可以预测是否会对业务造成影响。
- **检查要点**：检查随机数算法是否可以预测。

**CRYPTO-26: 椭圆曲线与密钥派生 (Elliptic Curve & KDF)**

- **描述**：区块链核心依赖椭圆曲线（如 secp256k1），需确保密钥派生过程符合标准，避免弱熵源导致的私钥可预测。

- **检查要点**：确保使用加密安全的伪随机数生成器（CSPRNG）。审查密钥派生函数（如 PBKDF2, Argon2, Scrypt）的迭代次数和盐值（Salt）长度，防止离线暴力破解。



**CRYPTO-27: 散列函数与哈希碰撞 (Hash Functions)**

- **描述**：确保在协议中使用抗碰撞的加密哈希算法，防止恶意构造的数据产生相同的摘要。

- **检查要点**：验证是否弃用了 MD5 或 SHA-1 等弱算法，统一使用 SHA-256 或 SHA-3。检查对于特定上下文（如 merkle tree 构建）是否加入了域分离（Domain Separation）前缀，防止哈希函数被跨上下文重用。


**CRYPTO-28: 加密与签名分离 (Encryption vs. Signing)**

- **描述**：区分加解密密钥与签名密钥，确保同一私钥不会被用于两种不同的数学逻辑，防止不同算法间的互操作性攻击。

- **检查要点**：审查代码逻辑，确保不存在用签名私钥进行数据解密的行为。如果必须使用同一密钥，应强制执行密钥分层或用途声明（Purpose Binding）。


**CRYPTO-29: 密钥销毁与内存清理 (Key Zeroing)**

- **描述**：敏感数据（如私钥、中间哈希值）在内存中残留可能被恶意进程 dump 出去。

- **检查要点**：审查内存申请与释放逻辑，确保敏感数据在处理完毕后通过 `memset` 或 `secure_zero` 清理内存。检查编译器是否会对“看起来无用”的清除代码进行优化（导致失效），应使用 `volatile` 指针或特定平台的安全清除函数。


**CRYPTO-30: 协议版本与算法协商 (Version & Algorithm Negotiation)**

- **描述**：防止协议向后兼容引发的“降级攻击”，强制要求使用当前最优的加密算法套件。

- **检查要点**：检查系统是否存在支持旧版本弱加密算法的配置路径。建立严格的白名单机制，拒绝任何不符合安全基准要求的加密协议握手或请求。
