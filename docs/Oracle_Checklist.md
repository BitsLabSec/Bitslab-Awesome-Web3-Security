- **ORC-01 是否使用了不易被操纵的预言机？**
    - **描述**：如果价格（或更广义的资产间汇率）来源于余额比率，则可能被操纵。闪电贷和捐赠攻击是常见的价格操纵手段。
    - **建议**：建议采用Chainlink预言机获取资产价格，并实现内部会计机制，而非依赖balanceOf计算价格。
  
- **ORC-02 预言机是否使用了已弃用的Chainlink函数？**
    - **描述**：使用latestRoundData()等已弃用的Chainlink函数可能返回过时或不正确的数据，影响智能合约的完整性。
    - **建议**：将已弃用的函数替换为当前推荐的方法，以确保从预言机准确获取数据。
  
- **ORC-03 返回的价格是否经过非零验证？**
    - **描述**：价格喂价可能返回零值，这种情况必须作为无效数据处理。
    - **建议**：确保返回的价格不为零。
  
- **ORC-04 是否验证了价格的更新时间？**
    - **描述**：未来某些价格喂价可能不再被支持。为确保使用准确的价格，必须定期检查最后更新时间戳是否超出预设的延迟阈值。
    - **建议**：实现一个机制来检查价格喂价的心跳，并将其与预设的最大延迟时间（MAX_DELAY）进行比较。根据观察到的实际心跳情况调整MAX_DELAY变量的取值。

- **ORC-05 是否有验证机制检查Rollup定序器是否在运行？**
    - **描述**：Rollup定序器可能离线，导致使用陈旧价格引发潜在漏洞。
    - **建议**：利用定序器运行状态喂价来确认定序器是否正常工作。

- **ORC-06 预言机的时间加权平均价格（TWAP）周期是否设置得当？**
    - **描述**：TWAP周期设置不当可能被利用来操纵价格。
    - **建议**：调整TWAP周期至一个既能降低价格操纵风险，又能提供及时价格更新的合理时长。

- **ORC-07 所需的喂价交易对是否在所有部署的链上都受支持？**
    - **描述**：在多链部署中，确保所需的喂价交易对在所有链上都可用且一致至关重要。
    - **建议**：检查所有链上支持的喂价交易对，确保它们保持一致。

- **ORC-08 价格喂价的心跳频率是否适合当前使用场景？**
    - **描述**：心跳频率过慢可能无法满足某些场景的需求。
    - **建议**：评估业务场景的实际需求，确保所选用的价格喂价心跳频率与之相匹配。
  
- **ORC-09 使用不同喂价时是否存在小数精度不一致的情况？**
    - **描述**：不同喂价可能具有不同的小数精度，这可能导致计算不准确。
    - **建议**：确保合约能够处理不同喂价源之间的小数精度差异。

- **ORC-10 喂价地址是否被硬编码？**
    - **描述**：硬编码的喂价地址可能带来问题，尤其是当地址被弃用或从一开始就不准确时。
    - **建议**：审查并验证硬编码的喂价地址。考虑引入未来如需更新地址的机制。

- **ORC-11 如果预言机价格更新被抢先交易会发生什么？**
    - **描述**：预言机价格更新可能被抢先交易，引发各种问题。
    - **建议**：确保协议在预言机价格更新被抢先交易的情况下不受影响。

- **ORC-12 系统如何处理预言机可能发生的回滚？**
    - **描述**：未预料到的预言机回滚可能导致拒绝服务。
    - **建议**：在预言机调用周围实现try/catch捕获机制，并准备备用策略。

- **ORC-13 协议是否使用了Balancer的预言机（getTimeWeightedAverage）？**
    - **描述**：Balancer池中的价格仅在发生交易（如兑换）时才会更新。由于更新频率不足，Balancer预言机提供的价格无法反映资产的真实价值。
    - **建议**：不要使用Balancer的预言机进行任何定价。
  
- **ORC-14 所使用的喂价是否适合底层资产？**
    - **描述**：为stETH使用ETH喂价，或为WBTC使用BTC喂价，可能会引入底层资产脱钩的风险。
    - **建议**：确保喂价能够准确代表底层资产，以应对潜在的脱钩风险。

- **ORC-15 合约是否易受预言机操纵影响，尤其是使用来自AMM的现货价格时？**
    - **描述**：依赖AMM现货价格作为预言机可能通过闪电贷被操纵。
    - **建议**：选择可靠且防篡改的预言机源。避免在未经额外检查的情况下直接使用来自AMM的现货价格。
  
- **ORC-16 系统如何处理闪崩期间可能出现的价格不准确？**
    - **描述**：在闪崩期间，预言机可能返回不准确的价格。
    - **建议**：实施检查机制，确保预言机返回的价格在预期范围内，以防范潜在的闪崩漏洞。

- **ORC-17 奖励的分发是否可能被不当延迟或提前领取？**
    - **描述**：对奖励分发时间的操纵可能对用户和协议预期的激励效果产生不利影响。
    - **建议**：对奖励分发实施时间控制和约束，以维持协议的预期行为。
  
- **ORC-18 系统如何处理Pyth预言机的价格时效性问题？**
    - **描述**：Pyth预言机采用拉取模型（pull model），需要使用者自行提供来自线下提供商的签名价格。这种方式无法保证当前使用的价格是最新的，这意味着攻击者可能在同一笔交易内，以一个基准价格进入流动性头寸，再以另一个价格退出，从而实现无风险套利。
    - **建议**：在每次使用Pyth预言机价格时，增加价格时效性验证机制。检查价格时间戳与当前区块时间的差值是否在可接受范围内（如MAX_AGE），确保使用的价格足够新鲜。同时，对于依赖价格进行头寸开平的关键操作，应限制同一交易内不允许价格跳跃过大，或强制要求两次操作使用同一价格更新。
  
- **ORC-19 协议如何处理支持单一Token对应多种资产时的预言机价格偏差？**
    - **描述**：当协议允许用户使用多种不同资产作为同一Token的抵押或存入时（例如多种稳定币都视为USD等价物），若未妥善管理这些资产之间的价格偏离，攻击者可在价格脱钩时，利用快速存取款机制进行套利，提取超出实际价值的资产。
    - **建议**：每种支持的资产使用独立的价格喂价，并在存取款时验证其与基准价值的偏差是否在安全范围内。若偏差超过预设阈值（如1%），应暂停该资产的存取操作，或应用惩罚性费率抵消套利空间。同时，对快速存取款操作增加时间锁或延迟机制，降低瞬时价格波动的影响。
  
- **ORC-20 系统如何处理外部预言机（如Uniswap V3）数据结构访问时的安全性？**
    - **描述**：当依赖外部预言机数据结构时，若未验证所访问数组元素是否已正确初始化且在有效范围内，特别是当这些数据结构可被外部参与者修改时，可能导致合约读取到未初始化或越界的数据，引发逻辑错误或漏洞。同时，时间加权平均计算若使用了预期时间范围外的数据点，将导致价格计算结果失真。
    - **建议**：在访问外部预言机数据结构前，始终验证数组索引是否在合法范围内，并确保目标元素已完成初始化。对于时间加权平均计算，应严格限定使用指定时间窗口内的数据点，并对窗口边界进行校验，防止因包含过期或未来数据导致计算偏差。
  
- **ORC-21 系统如何确保预言机或外部数据源在同一交易内不会提供不一致的数据？**
    - **描述**：当合约在同一交易的多次调用中从同一外部数据源获取数据时，若该数据源的状态可在交易中途被改变（例如通过另一个合约调用或抢先交易），可能导致前后获取的数据不一致，引发逻辑漏洞或被攻击者利用进行套利。
    - **建议**：在同一交易中首次获取外部数据后，将其缓存以供后续使用，确保整个交易期间使用一致的数据视图。对于关键操作，应考虑在交易开始时对依赖的外部状态进行快照，并基于该快照完成所有计算。
