# ERC721 - Checklist

## ERC721

一、 标准合规性基础

- **ERC721-01: 是否完整实现了ERC721标准接口？**
  - **描述**：**ERC721** 标准定义了必须实现的核心函数和事件，包括 `balanceOf`、`ownerOf`、`transferFrom`、`safeTransferFrom`（两种重载）、`approve`、`setApprovalForAll`、`getApproved`、`isApprovedForAll`，以及 `Transfer`、`Approval`、`ApprovalForAll` 事件。任何缺失都会破坏与钱包、市场等基础设施的兼容性。
  - **建议**：使用经过审计的标准库（如 `OpenZeppelin` 的 **ERC721** 实现），避免从零开始实现。确保所有必需函数和事件都存在且行为符合规范。

- **ERC721-02: 是否实现了ERC165接口检测？**
  - **描述**：**ERC721** 要求合约实现 **ERC165** 的 `supportsInterface` 函数，以便其他合约和工具能够查询该合约支持的接口。缺失此功能可能导致合约无法被正确识别为 NFT 合约。
  - **建议**：在合约中集成 **ERC165**，并在 `supportsInterface` 中返回 **ERC721** 接口 ID（`0x80ac58cd`）和 **ERC165** 自身接口 ID（`0x01ffc9a7`）。

- **ERC721-03: tokenURI的返回格式是否符合标准？**
  - **描述**：`tokenURI` 函数应返回指向元数据 JSON 文件的 URI，该 JSON 文件需要符合市场（如 **OpenSea**）期望的格式（包含 `name`、`description`、`image` 等字段）。如果格式不正确，NFT 可能无法在市场上正确显示。
  - **建议**：遵循通用元数据规范（如 **OpenSea** 元数据标准），返回指向 **IPFS**、**Arweave** 等去中心化存储的 URI，并确保 JSON 结构完整。

- **ERC721-04: 是否正确处理了零地址转账？**
  - **描述**：向零地址（`0x0`）转账意味着永久销毁代币。**ERC721** 标准未强制禁止向零地址转账，但这可能导致意外销毁或需要专门的销毁机制。
  - **建议**：在 `transferFrom` 和 `safeTransferFrom` 中增加 `require(to != address(0), "ERC721: transfer to zero address")` 的检查。如需销毁功能，应实现显式的 `burn` 函数。


- **ERC721-05: 构造函数是否正确定义？**
  - **描述**：在 `Solidity` 旧版本中，构造函数名必须与合约名完全一致；`0.4.22` 引入 `constructor` 关键字后，错误地写成 `function constructor()` 会使其变成普通函数，任何人都可调用。
  - **建议**：使用 `Solidity` `0.8.x` 版本，并确保构造函数使用 `constructor` 关键字正确声明，避免权限被意外公开。

二、 安全转移与回调机制

- **ERC721-06: 是否使用了safeTransferFrom而非transferFrom？**
  - **描述**：`transferFrom` 直接将代币发送给目标地址。如果目标地址是合约且未实现转出方法，代币将被永久锁定在该合约中。
  - **建议**：优先使用 `safeTransferFrom`，它会在发送给合约时检查接收方是否实现了 `onERC721Received` 函数，防止NFT意外丢失。

- **ERC721-07: 是否防范了safeTransferFrom回调导致的重入攻击？**
  - **描述**：`safeTransferFrom` 会调用接收方合约的 `onERC721Received` 函数。如果接收方合约是恶意的，它可以在该回调中重入调用发起方的函数，利用状态未更新的漏洞进行攻击（如多次铸造、重复领取等）。
  - **建议**：在调用 `safeTransferFrom` 之前完成所有状态更新（遵循检查-生效-交互模式），并为涉及 NFT 转移的关键函数添加 `nonReentrant` 修饰符。

- **ERC721-08: 批量铸造时是否考虑了循环中的重入风险？**
  - **描述**：在批量铸造的循环中使用 `_safeMint`，每次铸造都会调用接收方的 `onERC721Received`。如果攻击者合约在回调中再次进入铸造循环，可能利用 `totalSupply` 等状态变量未及时更新的时间差，铸造超过限制数量的代币。
  - **建议**：在批量铸造/转移前预计算并检查总量，在循环开始前完成所有验证。为整个批量操作添加重入锁，或在循环期间禁止状态重入。

- **ERC721-9: 是否检查了onERC721Received的返回值？**
  - **描述**：`safeTransferFrom` 要求接收方合约返回特定的魔数（`0x150b7a02`）以确认正确实现了接收函数。如果返回错误值或未正确实现，NFT可能因无法转移而DOS。
  - **建议**：使用 `OpenZeppelin` 的 `_checkOnERC721Received` 函数或在实现中确保正确验证返回值。

- **ERC721-10: 是否区分了对EOA和合约地址的转移处理？**
  - **描述**：`safeTransferFrom` 对 EOA 地址不会触发回调，但对合约地址必须进行接收能力检查。错误地将所有地址都当作合约处理，或相反，都可能导致问题。
  - **建议**：使用 `OpenZeppelin` 的 `Address.isContract()` 函数判断目标地址是否为合约，仅在必要时进行回调检查。*注意：`Address.isContract()`检查可能会被绕过*

三、 授权与权限控制

- **ERC721-11: approve授权是否可能被滥用？**
  - **描述**：`approve` 函数允许所有者授权某个地址管理单个 NFT。如果用户误将 NFT 授权给恶意合约，攻击者可立即调用 `transferFrom` 转走该 NFT。
  - **建议**：在前端清晰展示授权信息，包括合约名称、地址和授权范围。考虑使用 **ERC-4494** 签名授权（Permit）进行无 gas 授权，同时提高安全性。

- **ERC721-12: setApprovalForAll是否存在过度授权风险？**
  - **描述**：`setApprovalForAll` 允许用户授权一个运营商管理其所有 NFT。这是一个强大的权限，一旦授权，如果运营商合约存在漏洞或被攻击，用户的所有 NFT 都将面临风险。
  - **建议**：向用户明确提示 `setApprovalForAll` 的广泛权限。建议用户在使用后及时撤销不必要的授权，并在钱包中定期检查已授权列表。


- **ERC721-13: 是否实现了EIP-4494（Permit for ERC721）？**
  - **描述**：**EIP-4494** 允许通过签名进行授权，实现无 gas 授权和更好的用户体验。用户无需发送交易即可授权，签名可以被中继者提交。
  - **建议**：如果项目需要频繁授权操作，考虑集成 **EIP-4494**，但需注意签名验证的安全实现，包括防止重放攻击和过期检查。

- **ERC721-14: 授权和批准是否在状态更新后正确重置？**
  - **描述**：当 NFT 被转移后，原有授权应自动清除，防止旧持有者或已撤销的授权者继续使用。标准中 `transferFrom` 应清除对当前 `tokenId` 的授权。
  - **建议**：在转移函数中明确将 `_tokenApprovals[tokenId]` 设为零地址（`0x0`），并更新所有者和余额状态。

四、 元数据处理

- **ERC721-15: tokenURI的存储方案是否考虑了永久性？**
  - **描述**：如果使用 HTTP 链接存储元数据，一旦服务器宕机或链接失效，NFT 的图片和属性将无法访问，导致资产价值受损。
  - **建议**：使用 **IPFS**（内容寻址）或 **Arweave**（永久存储）存储元数据和媒体文件，确保即使原始上传者离线，内容仍可访问。

- **ERC721-16: 元数据更新时是否通知了市场？**
  - **描述**：当 NFT 元数据发生变化（如动态 NFT、游戏属性更新）时，市场索引器需要知道刷新缓存。没有通知机制，用户看到的是过时的信息。
  - **建议**：实现 **ERC-4906**（Metadata Update 事件），在元数据变更时发出 `MetadataUpdate` 或 `BatchMetadataUpdate` 事件，通知市场刷新。


- **ERC721-17: 元数据JSON是否符合预期格式？**
  - **描述**：市场的索引器期望特定格式的元数据 JSON，包括 `name`、`description`、`image`、`attributes` 等字段。格式错误可能导致 NFT 在市场无法显示。
  - **建议**：遵循 **OpenSea** 等主流市场的元数据标准，在 JSON 中包含标准字段，并确保 `image` 或 `image_url` 指向有效的媒体资源。

- **ERC721-18: 是否存在元数据与代币ID的映射漏洞？**
  - **描述**：某些实现中，元数据 URI 可能由代币 ID 动态生成（如 `string(abi.encodePacked(baseURI, tokenId.toString(), ".json"))`）。如果验证不当，攻击者可能访问未铸造代币的元数据或利用路径遍历漏洞。
  - **建议**：验证 `tokenId` 的有效性（如存在性检查），确保 `baseURI` 不以危险字符结尾，避免路径遍历。

五、 扩展功能与标准集成

- **ERC721-19: 是否实现了版税标准（ERC-2981）？**
  - **描述**：**ERC-2981** 为 NFT 定义了链上版税信息标准，市场可以据此向创作者支付二级销售版税。没有此标准，版税信息需要链下约定，执行不一致。
  - **建议**：实现 **ERC-2981** 接口，在 `royaltyInfo` 函数中返回版税接收地址和费率。注意不同市场对版税的支持程度可能不同。

- **ERC721-20: 是否实现了可租用角色（ERC-4907）？**
  - **描述**：**ERC-4907** 引入带过期时间的“用户”角色，允许 NFT 所有者将使用权临时授予他人，而无需转移所有权。这对游戏资产租赁、订阅服务等场景很有价值。
  - **建议**：如果项目涉及 NFT 租借或使用权分离，考虑集成 **ERC-4907**，并确保与现有授权逻辑兼容。

- **ERC721-21: 是否支持代币绑定账户（ERC-6551）？**
  - **描述**：**ERC-6551** 允许每个 NFT 拥有自己的智能合约账户（TBA），可以持有资产、与其他协议交互，使 NFT 从静态收藏品变成可编程实体。
  - **建议**：如果项目需要 NFT 具备复杂行为（如游戏角色持有装备、身份聚合等），研究并集成 **ERC-6551**，但需注意额外的安全考量。

- **ERC721-22: 批量铸造是否实现了Gas优化（如ERC-721A）？**
  - **描述**：标准 **ERC721** 实现中，批量铸造 `N` 个 NFT 需要 `N` 次存储写入，Gas 成本呈线性增长。**ERC-721A** 通过连续 ID 和所有权打包，使批量铸造的 Gas 成本接近单次铸造。
  - **建议**：对于需要高频批量铸造的场景（如 PFP 项目、游戏资产发放），考虑使用经过审计的 **ERC-721A** 实现。

- **ERC721-23: 是否支持批量事件（ERC-2309）？**
  - **描述**：**ERC-2309** 定义了 `ConsecutiveTransfer` 事件，可以在一次事件中表示连续多个代币的铸造或转移，大幅减少事件日志 Gas 成本。
  - **建议**：在合约初始化铸造或大规模空投时，使用 `ConsecutiveTransfer` 事件优化 Gas 消耗，但需确保市场索引器支持该事件解析。

- **ERC721-24: 是否集成了权限控制机制（如ERC-721C）？**
  - **描述**：**ERC-721C** 允许合约配置白名单、授权规则和转移安全级别，限制哪些市场或合约可以操作代币，增强创作者控制力。
  - **建议**：如果项目需要对二级市场交易施加限制（如仅特定市场可交易），考虑集成 **ERC-721C** 的 `TransferValidator` 机制，并正确配置安全级别。

六、 铸币与销毁逻辑

- **ERC721-25: 铸币函数是否有正确的访问控制？**
  - **描述**：铸币（`mint`）函数应仅限授权角色（所有者、团队地址、通过验证的用户）调用，否则任何人都可无限铸造，导致NFT价值归零。
  - **建议**：为铸币函数添加 `onlyOwner` 或基于角色的访问控制（如 `OpenZeppelin` 的 `AccessControl`）。对于公开铸币，实现签名验证或 Merkle 证明验证。

- **ERC721-26: 铸币数量是否受到合理限制？**
  - **描述**：没有单次铸币上限或总供应量限制的合约，可能导致 Gas 消耗过高的交易失败，或被攻击者一次铸造大量代币冲垮市场。
  - **建议**：实现每地址铸币上限、单次交易上限和总供应量上限，并在函数中验证这些限制。

- **ERC721-27: 销毁函数是否正确更新状态？**
  - **描述**：`burn` 函数应从所有者余额中减去代币，从所有者NFT列表中移除，并清除授权信息。实现不完整可能导致余额与所有权不一致。
  - **建议**：使用标准库的 `_burn` 函数，确保在销毁时触发 `Transfer` 事件（`from` 为原所有者，`to` 为零地址），并清除所有相关状态。

- **ERC721-28: 销毁时是否检查了调用者权限？**
  - **描述**：销毁函数应仅允许代币所有者或已授权地址调用。权限检查缺失可能导致任何人销毁他人的 NFT。
  - **建议**：在销毁前验证 `_isApprovedOrOwner(_msgSender(), tokenId)`，确保只有授权方可销毁。

- **ERC721-29: 总供应量（totalSupply）是否保持一致性？**
  - **描述**：在铸币和销毁操作中，`totalSupply` 必须相应增加或减少。如果回调导致状态不一致，可能影响依赖 `totalSupply` 的其他合约逻辑。
  - **建议**：在状态变更前后确保 `totalSupply` 正确更新。在批量操作中，注意回调可能引起的临时状态不一致。

七、 枚举与可查询性

- **ERC721-30: 是否实现了可枚举扩展（ERC721Enumerable）？**
  - **描述**：`ERC721Enumerable` 允许通过索引查询所有代币或特定所有者的代币列表。这对于需要遍历代币的应用（如市场、游戏）很重要，但会增加 Gas 成本。
  - **建议**：如果应用需要枚举功能，可集成 `ERC721Enumerable`，但需注意它在大量代币场景下的 Gas 消耗。也可考虑链下索引替代。

- **ERC721-31: 枚举实现是否存在Gas漏洞？**
  - **描述**：如果枚举实现中涉及大规模循环（如 `tokenOfOwnerByIndex` 内部），可能因 Gas 消耗过高导致交易失败或 DOS 攻击。
  - **建议**：审查枚举相关函数的实现，确保没有无限制的循环。对于所有者代币列表，可考虑使用映射加数组的结构，但需注意更新时的成本。

- **ERC721-32: 是否存在可被操纵的ownerOf查询？**
  - **描述**：`ownerOf` 应返回当前所有者，在销毁后调用应回滚。某些实现可能返回已销毁代币的原所有者或被操纵的数据。
  - **建议**：确保 `ownerOf` 对不存在或已销毁的代币 ID 进行 `revert` 检查。在状态变化时正确更新所有权映射。

- **ERC721-33: 余额查询（balanceOf）是否考虑了所有代币？**
  - **描述**：`balanceOf` 应返回地址当前持有的所有代币数量，包括通过转移和铸造获得的。错误实现可能漏计或重复计数。
  - **建议**：使用映射 `_balances` 跟踪每个地址的NFT数量，并在每次转移、铸造、销毁时正确更新。

八、 外部集成与互操作性

- **ERC721-34: 与其他合约交互时是否使用了SafeERC721？**
  - **描述**：与 **ERC721** 合约交互时，直接调用 `transferFrom` 可能忽略对接收方合约的检查。使用安全转移函数可避免代币锁定。
  - **建议**：在与外部 **ERC721** 交互时，优先调用 `safeTransferFrom`，并确保接收方合约正确处理回调。

- **ERC721-35: 是否处理了重入攻击防护？**
  - **描述**：如前述，`safeTransferFrom` 的回调可能成为重入攻击的入口。缺少重入锁可能导致状态不一致。
  - **建议**：为涉及 NFT 转移的关键函数添加 `nonReentrant` 修饰符，尤其是在函数内存在多个外部调用或状态更新的情况下。

- **ERC721-36: 是否考虑了L2部署的特殊性？**
  - **描述**：在 Layer 2（如 **Arbitrum**、**Optimism**）上部署 **ERC721**，需要考虑桥接行为、Sequencer 最终性和 Gas 经济性差异。
  - **建议**：部署前研究目标 L2 的 NFT 生态系统、市场索引方式和桥接机制。在主网和 L2 上测试 Gas 消耗和最终性时间。

- **ERC721-37: 是否考虑了跨链元数据一致性？**
  - **描述**：当 NFT 跨链桥接时，不同链上的元数据可能不一致，导致持有者困惑或价值错配。
  - **建议**：使用 **CAIP-2** 链标识符区分不同链的变体，并考虑采用可验证的链下解析方案（如 **EIP-3668**）保持跨链一致性。

九、 管理功能与权限

- **ERC721-38: 管理员转移是否是两步骤流程？**
  - **描述**：单步所有权转移极易因输错地址导致合约永远失去管理权限。如果合约包含暂停、铸币等特权功能，这可能造成严重后果。
  - **建议**：实现两阶段所有权转移：先提名新所有者，再由其接受。可使用 `OpenZeppelin` 的 `Ownable2Step` 合约。

- **ERC721-39: 基础URI（baseURI）设置是否有访问控制？**
  - **描述**：`baseURI` 通常用于所有代币的元数据前缀。如果没有访问控制，任何人都可修改它，使所有代币的元数据指向恶意或无效内容。
  - **建议**：`setBaseURI` 函数应仅限所有者调用，并考虑使用时间锁或多签地址来管理此类敏感变更。

- **ERC721-40: 合约是否可升级？升级机制是否安全？**
  - **描述**：可升级合约允许修复 bug 和添加功能，但如果升级逻辑实现不当（如代理选择错误、初始化函数可重入），可能导致资金损失或永久锁仓。
  - **建议**：使用经过审计的代理模式（如 **UUPS**、透明代理），并确保初始化函数只能调用一次。升级前进行充分测试并考虑时间锁。

- **ERC721-41: 是否存在自毁（selfdestruct）函数？**
  - **描述**：包含 `selfdestruct` 的合约可以被所有者销毁，所有以太币和代码将被移除，导致 NFT 永久无法访问（但代币数据仍存在于链上）。
  - **建议**：避免在合约中包含 `selfdestruct`，除非有明确的业务需求和恢复计划。

- **ERC721-42: 暂停机制是否实现了？**
  - **描述**：在紧急情况下，暂停 NFT 转移可以保护用户资产，但如果暂停功能被滥用或缺乏访问控制，也可能成为作恶工具。
  - **建议**：实现带访问控制的暂停机制（如 `OpenZeppelin` 的 `Pausable`），并在关键函数中添加 `whenNotPaused` 修饰符。

十、 签名与链下授权

- **ERC721-43: 签名验证是否使用了EIP-712？**
  - **描述**：使用 `eth_sign` 或 `personal_sign` 签署的消息缺乏结构化，容易被钓鱼或重放攻击。**EIP-712** 提供了结构化、可读的签名数据格式。
  - **建议**：在需要链下签名的场景（如白名单铸币、授权挂单）中使用 **EIP-712**，确保域分隔、包含 nonce 和截止时间，并在合约中正确验证。

- **ERC721-44: 签名验证是否包含了nonce和截止时间？**
  - **描述**：没有 nonce 的签名可被重复使用；没有截止时间的签名可被无限期延迟使用。这可能导致签名被重放攻击利用。
  - **建议**：在签名内容中包含 nonce（在合约中跟踪使用状态）和截止时间（deadline），并在验证时检查两者。

- **ERC721-45: 签名验证函数是否正确恢复了签名者？**
  - **描述**：`ecrecover` 可能返回零地址而不报错，且易受签名可锻性攻击。直接使用 `ecrecover` 而不验证地址有效性可能导致攻击者伪造签名。
  - **建议**：使用 `OpenZeppelin` 的 `ECDSA` 库进行签名恢复和验证，并在验证后确保恢复的地址非零且符合预期。

- **ERC721-46: 前端是否清晰展示了签名内容？**
  - **描述**：用户可能在不知情的情况下签署恶意交易，例如签署了授权攻击者转移 NFT 的消息。前端显示模糊或误导性信息是常见钓鱼手段。
  - **建议**：在前端使用结构化签名显示，明确告知用户签名内容、用途和潜在风险。对于高价值操作，要求用户使用硬件钱包进行签名。

- **ERC721-47: 是否验证了签名者的授权状态？**
  - **描述**：某些操作（如使用签名铸造）需要验证签名者是否为特定角色或白名单地址。仅验证签名有效性不足以确保权限正确。
  - **建议**：在验证签名后，检查签名者地址是否在白名单中、是否拥有必要角色或满足其他业务条件。

十一、 其他安全检查

- **ERC721-48: 合约是否防范了整数溢出/下溢？**
  - **描述**：`Solidity` `0.8.x` 之前，整数运算可能静默溢出。例如，余额计数可能绕回导致不正确结果。
  - **建议**：使用 `Solidity` `0.8.x` 及以上版本（内置溢出检查）或使用 `SafeMath` 库进行所有算术运算。

- **ERC721-49: 函数可见性和状态变量作用域是否正确？**
  - **描述**：将本应 `internal` 或 `private` 的函数错误声明为 `public` 可能导致任何人调用敏感逻辑。公开映射可能暴露数据。
  - **建议**：全面审查函数和状态变量的可见性，遵循最小权限原则。对敏感函数使用适当的访问控制。

- **ERC721-50: 是否存在因循环导致的Gas陷阱？**
  - **描述**：需要遍历大量代币或用户的函数（如批量转移、批量销毁），如果循环无上限，可能因超过区块 Gas 限制而永远无法执行。
  - **建议**：限制批量操作的单次数量，或采用“拉取模式”让用户自行领取/操作，而非合约主动推送。

- **ERC721-51: 合约初始化函数是否可被多次调用？**
  - **描述**：在可升级合约中，初始化函数通常代替构造函数。如果初始化函数缺少防重入保护，攻击者可多次调用重新初始化合约，篡改关键参数。
  - **建议**：使用 `OpenZeppelin` 的 `Initializable` 合约，为初始化函数添加 `initializer` 修饰符，确保只能调用一次。

- **ERC721-52: 是否包含了足够的链下监控和事件？**
  - **描述**：关键操作（铸币、转移、授权、管理变更）应触发标准事件，以便链下监控系统追踪异常活动和维护数据一致性。
  - **建议**：确保所有状态变化都触发相应事件，包括标准 `Transfer`、`Approval` 系列事件，以及自定义的业务事件。
