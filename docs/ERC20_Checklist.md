
# ERC20 - Checklist
 
## ERC20
一、 标准合规性检查
- **ERC20-01: 是否遵循了函数返回值的规范？**
    - **描述**：ERC20标准中，`transfer`、`approve`和`transferFrom`函数应该返回一个布尔值（bool）来表示成功或失败。但一些知名代币（如**USDT**）并未实现返回值，这会导致依赖返回值进行判断的合约交互失败。
    - **检查要点**：确保你的合约遵循标准返回bool值。如果是在合约中与外部ERC20交互，应使用 `OpenZeppelin` 的 `SafeERC20` 库来统一处理有无返回值的情况。

- **ERC20-02: 转账和授权操作是否触发了相应的事件？**
    - **描述**：标准要求任何改变状态的转账（`transfer`）和授权（`approve`）操作必须触发`Transfer`和`Approval`事件。缺失事件会影响链下服务（如区块浏览器、监控系统）对交易状态的追踪。
    - **检查要点**：在`transfer`、`transferFrom`和`_mint`、`_burn`等内部函数中触发`Transfer`事件；在`approve`函数中触发`Approval`事件。

- **ERC20-03: 余额不足或授权不足时，交易是否回滚？**

    - **描述:** 当发送者余额不足或授权额度不足时，合约必须通过`require`或`revert`使交易失败并回滚所有状态更改，而不是简单地返回`false`。使用`if`判断并返回`false`可能导致"假充值"漏洞。
    - **检查要点：** 在`transfer`、`transferFrom`和`_burn`等函数中，使用`require(balanceOf[account] >= amount)`来验证余额。

- **ERC20-04: 代币的基本信息（名称、符号、小数位数）是否正确且可访问？**

    - **描述:** `name()`、`symbol()`和`decimals()`应正确地返回值。`decimals()`通常返回18，但不是强制标准，像**USDC**返回`6`。另外，一些早期代币可能使用`bytes32`而非`string`返回名称和符号，导致调用失败。
    - **检查要点：** 查询函数应能稳定、正确地返回值。与外部代币交互时，需考虑处理`string`与`bytes32`的兼容性问题。

二、 核心安全漏洞防护
- **ERC20-05: 是否防范了"授权竞态攻击"？**

    - **描述:** 这是ERC20最著名的漏洞之一。如果用户将授权从`N`改为`M`，恶意`spender`可以抢先在第一个`approve`交易确认前使用旧的`N`额度，然后在`approve`确认后又获得`M`额度，从而总共花费`N+M`代币。
    - **检查要点：** 避免直接覆盖非零授权。使用 `OpenZeppelin` 的 `increaseAllowance / decreaseAllowance`，或在修改授权前先将额度设置为`0`。

- **ERC20-06: 是否防范了"构造函数错误"？**

    - **描述:** 在`Solidity` `0.4.22`版本之前，构造函数名必须与合约名完全一致（包括大小写）。拼写错误会导致构造函数变成可公开调用的普通函数，攻击者可调用它来篡改合约所有者权限。在`0.4.22`引入`constructor`关键字后，错误地写成`function constructor()`同样会使其变为普通函数。
    - **检查要点：** 确保构造函数正确声明。对于旧版本合约，重点检查构造函数名是否与合约名完全匹配。

- **ERC20-07: 关键函数是否有正确的访问控制？**

    - **描述:** 涉及铸币、烧毁、设置白名单、暂停转账等敏感操作的函数，必须使用`onlyOwner`或自定义的角色权限控制。公开这些函数将使任何用户都能调用，导致严重安全风险。
    - **检查要点：** 对所有管理职能（如铸币、暂停、设置费用等）添加明确的访问控制修饰符。

- **ERC20-08: 转账目标地址是否为无效地址？**

    - **描述:** 应禁止向零地址（`0x0`）转账，因为大量代币被发送到黑洞地址会导致永久丢失。也建议禁止向代币本身的合约地址转账，这是一个常见的用户操作失误。
    - **检查要点：** 在`transfer`和`transferFrom`中增加`require(to != address(0))`以及`require(to != address(this))`的检查。

三、 外部交互与兼容性
- **ERC20-09: 在与非标准ERC20代币交互时，是否使用了`SafeERC20`？**

    - **描述:** 如**USDT**、**BNB**等代币的`transfer`/`transferFrom`函数没有返回值。如果使用标准的`IERC20(token).transfer()`直接调用，Solidity编译器会检查返回值，但因没有返回值而导致交易回滚。
    - **检查要点：** 在与外部代币交互的合约中，始终引入并使用 `OpenZeppelin` 的 `SafeERC20` 库，它通过底层调用并手动解码返回值来处理这类非标准情况。

- **ERC20-10: 转账金额是否检查了零值？**

    - **描述:** 大多数ERC20代币允许转账`0`个代币。但少数代币（如**LEND**）在转账金额为`0`时会直接回滚交易。如果你的协议逻辑未考虑到这一点，可能在集成这些代币时遭遇意外的拒绝服务。
    - **检查要点：** 在调用外部代币的`transfer`或`transferFrom`前，检查转账金额是否大于`0`，或对零值进行无操作处理。

- **ERC20-11: 是否考虑了通缩型代币或具备转账钩子的代币？**

    - **描述:** 一些代币在转账时会扣除手续费（即"通缩代币"），导致实际到账金额小于发送金额。还有的代币（如`ERC777`）在转账时会回调接收方合约，这可能导致重入攻击。
    - **检查要点：** 对于需要精确记账的协议，不应假设"转入数量=到账数量"。可以在转账前后通过`balanceOf`对比实际余额变化。对于有钩子的代币，确保协议有重入锁保护。

- **ERC20-12: 代币余额是否可能在不发生转账的情况下变化？**

    - **描述:** 如`stETH`（`Lido`）或`aToken`（`Aave`）等生息资产，其持有者余额会随时间推移而自动增加（即"rebasing"）。如果协议内部记录了用户的存款数量，并与实际代币余额不一致，可能导致收益分配错误。
    - **检查要点：** 如果协议不支持rebasing代币，应在文档和UI中明确警告用户，或要求用户存入封装版本（如`wstETH`）。

- **ERC20-13: 是否兼容高精度授权限制的代币？**

    - **描述:** 像**UNI**或**COMP**等代币，如果授权的金额大于$2^{96}$，交易会直接回滚。这对于习惯授权"无限大"（`type(uint256).max`）的用户来说是个陷阱。
    - **检查要点：** 在需要授权大额度的场景下，可以检测代币地址，对这些特定代币使用特定的较小额度，或使用逐步增加的授权方式。

- **ERC20-14: 是否考虑了代币的黑名单机制？**

    - **描述:** 一些合规性代币（如**USDC**）或具备反洗钱功能的代币，内置了地址黑名单。被列入黑名单的地址将无法发送或接收代币。
    - **检查要点：** 如果你的协议依赖资金锁定或特定地址的转账功能，需考虑黑名单导致交易失败的可能性，并在产品交互上做好提示。

- **ERC20-15: 对小数点后位数非18的代币处理是否正确？**

    - **描述:** 虽然`18`是常见选择，但许多主流代币并非如此，如**USDC**（6位）、**USDT**（6位）、**WBTC**（8位）。在前端展示或合约内部做乘除计算时，硬编码`10**18`将导致金额严重错误。
    - **检查要点：** 始终通过`decimals()`函数动态获取代币精度，并在UI和合约计算中使用该精度进行缩放。

四、 逻辑与权限管理
- **ERC20-16: 是否存在可能永久锁定资金的"自毁"或迁移函数？**

    - **描述:** 合约若包含`selfdestruct`操作码，或允许所有者将合约逻辑升级到一个无法访问的地址，都可能导致代币永久锁仓。
    - **检查要点：** 谨慎使用`selfdestruct`。如果合约可升级，建议采用两阶段所有权转移（即先提名，后接受）和有时间锁的升级流程，以防止单点故障和恶意操作。

- **ERC20-17: 铸币和烧币逻辑是否正确且受到保护**？

    - **描述:** 铸币应增加`totalSupply`和指定地址的余额，并触发`Transfer`事件（通常`from`地址为零地址）。烧币则应减少`totalSupply`和余额，触发`Transfer`事件（`to`地址为零地址）。这些功能通常应仅限所有者或特定的权限合约调用。
    - **检查要点：** 检查铸币/烧币函数的访问权限。验证供应量的变化与余额变化的一致性。

- **ERC20-18: 是否存在可以被操纵的"捐赠"或"直接转账"漏洞？**

    - **描述:** 如你之前关注的首位存款者攻击。在需要份额计算的合约（如LP池）中，攻击者可通过极小份额的铸造配合直接向合约转账来操纵汇率，导致后续用户份额归零。
    - **检查要点：** 在首次存款时，强制铸造一定数量的份额（如1000 `wei`）给死地址，以提高攻击者操纵汇率的成本。

- **ERC20-19: 合约所有权转移是否是两步骤流程？**

    - **描述:** 单步所有权转移非常危险。如果所有者不小心将所有权转移到了错误的地址（如错误的地址或丢失的私钥），合约将永远失去管理者权限。
    - **检查要点：** 实现为`transferOwnership(address _newOwner)`提名新所有者，并由新所有者调用`acceptOwnership()`来完成转移。

- **ERC20-20: 函数的可见性和状态变量的作用域是否正确？**

    - **描述:** 将本应是`internal`或`private`的关键函数（如内部铸币逻辑）错误地声明为`public`，可能导致任何人调用。映射或数组的公开可见性虽不影响安全，但可能无意中暴露数据。
    - **检查要点：** 全面审查函数和状态变量的可见性，确保没有不必要的公开暴露。

- **ERC20-21: 是否存在因循环或复杂逻辑导致的Gas陷阱？**

    - **描述:** 在需要遍历大量用户的函数（如空投领取、批量转账）中，如果循环无上限或单次操作Gas成本过高，可能导致函数执行时超过区块Gas限制而永远无法成功。
    - **检查要点：** 限制批量操作的规模，或设计成允许用户自行领取（拉取模式）而非主动分发（推送模式）。

- **ERC20-22: 数学运算是否考虑了溢出/下溢？**

    - **描述:** 在Solidity `0.8.x`版本之前，不加检查的整数运算会静默溢出。例如，余额为1，减去2，结果会变成$2^{256}-1$，而不是回滚。
    - **检查要点：** 使用Solidity `0.8.x`或更高版本（自带溢出检查），或使用`OpenZeppelin`的`SafeMath`库进行所有算术运算。

- **ERC20-23: 合约是否实现了防重入保护？**

    - **描述:** 虽然ERC20本身的转账通常不包含外部调用，但如果你的代币包含钩子函数（如ERC777），或在你的协议合约中进行了多次外部调用，就可能面临重入攻击风险。
    - **检查要点：** 在涉及多步操作（尤其是先更新状态后转账）的函数中，考虑使用`OpenZeppelin`的`ReentrancyGuard`。

- **ERC20-24: 重写的转账函数，转账时是否允许 from 和 to 地址相同？**

    - **描述:** 如果 from 和 to 地址相同，转账函数不能正确处理余额，可能会导致余额错误。
    - **检查要点：** from 和 to 地址相同，则不应在转账中减少或增加余额，以防止余额和余额跟踪错误。

- **ERC20-25: 重写的 `transferFrom()`函数，是否正确处理了代理的`approve`额度？**

    - **描述:** 如果重写的`transferFrom()`函数，没有处理代理的`approve`额度，代理可以超额花费用户的代币。
    - **检查要点：** 重写的`transferFrom()`函数中，应该及时更新代理的`approve`额度。
