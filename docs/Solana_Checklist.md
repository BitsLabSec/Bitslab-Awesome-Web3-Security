# Solana Checklist

## 一、 权限控制与身份认证 

- Sol-01: 是否在全局配置初始化时防范了“抢跑”夺权？
    - 描述：程序的初始化函数（如设置 Admin 或配置参数）如果没有权限限制或未在部署交易中原子化执行，攻击者可能先于开发者调用该函数，从而夺取程序的最高控制权。
    - 建议：在初始化函数中增加对调用者（Signer）的硬编码校验，或利用 Anchor 框架的原子化初始化机制，确保只有部署者能触发。

- Sol-02: 程序派生地址 (PDA) 的派生种子 (Seeds) 是否进行了严格校验？
    - 描述：如果仅校验 PDA 的 Owner 而不校验派生它的具体 Seeds，攻击者可以传入一个由当前程序派生的、但属于其他业务逻辑的 PDA 账户，从而实现数据伪造。
    - 建议：在验证 PDA 时，必须通过 `find_program_address` 重新计算并对比种子路径，确保传入的账户是当前特定业务逻辑下唯一的 PDA。

- Sol-03: 是否对 PDA 的归属权进行了明确验证？
    - 描述：攻击者可能会传入一个结构相同、由攻击者控制的程序派生的 PDA。如果合约只检查结构而不检查 `owner`，则会产生权限越权。
    - 建议：强制校验 PDA 账户的 `owner` 必须是当前的 Program ID。

- Sol-04: 特权角色（如 Admin）的变更是否采用了双步确认 (Two-Step) 机制？
    - 描述：单步转移权限（直接设置新地址）存在极高的容错风险，一旦填错地址，程序将永久失控。
    - 建议：采用“提议 (Propose) + 接受 (Accept)”模式。原管理员先提议新地址，新地址必须由其私钥签名调用接受函数后，权限转移才正式生效。

- Sol-05: 敏感辅助账户（如配置账户）的销毁权限是否受限？
    - 描述：如果 `close_account` 或类似的销毁逻辑缺乏权限校验，攻击者可能恶意注销核心配置账户，导致程序逻辑因找不到状态而陷入拒绝服务 (DoS)。
    - 建议：限制只有管理员或特定业务状态（如项目结束）下才能触发账户销毁，并校验目标地址的合法性。

- Sol-06: 协议升级或国库提款等高危操作是否强制要求多签 (Multisig)？
    - 描述：单一地址控制核心权限存在单点故障风险（私钥泄露或管理员作恶）。
    - 建议：将 Upgrade Authority 或 Treasury 权限交给 Squads 等多签程序管理，确保决策经过多数共识。

- Sol-07: 重要参数修改是否设置了时间锁 (Timelock) 缓冲期？
    - 描述：管理员若能瞬间修改费率、滑点或清算参数，用户将没有时间对不利改动做出反应（如撤资）。
    - 建议：对敏感参数修改设置延迟生效机制，通过事件告知用户，预留出足够的退出窗口。

- Sol-08: 敏感业务执行者的身份 (Signer) 是否具有唯一性确认？
    - 描述：某些业务逻辑（如修改用户个人资料）若未校验 `is_signer`，攻击者可以通过传入受害者的账户地址来伪造其身份。
    - 建议：在逻辑入口处严格校验核心操作者必须是当前交易的签名者（Anchor 中使用 `Signer<'info>` 类型）。

- Sol-09: 逻辑设计中是否规避了单一地址的绝对控制权？
    - 描述：若逻辑中存在某个地址可以任意重置全场数据，该点将成为黑客攻击的首选目标。
    - 建议：尽量去中心化管理逻辑，或将控制权分解到不同的模块中，防止单点泄露导致全盘崩溃。

- Sol-10: 治理权限是否存在“合法 Rug”风险？
    - 描述：管理员权力如果涉及修改用户已锁定的资产（如强行转移质押中的代币），则构成潜在的信任危机。
    - 建议：代码逻辑上应锁定用户既得利益相关的关键变量，管理员仅能修改全局费率等非资产归属类参数。

## 二、 资产安全与代币逻辑

- Sol-11: 代币 Mint 地址是否进行了严格的唯一性校验？
    - 描述：在兑换或质押业务中，如果合约不检查用户存入代币的 Mint 地址，攻击者可以发行一种“伪币”并存入，从而换取合约中的真币。
    - 建议：在状态中存储合法的 Mint 地址，并在所有资金入账操作中强制校验 `account.mint == expected_mint`。

- Sol-12: 关联代币账户 (ATA) 的归属权是否校验？
    - 描述：攻击者可能传入一个由其控制的、但属于受害者 Mint 的代币账户。若不校验 ATA 的 Owner，资金可能会被转错方向。
    - 建议：校验代币账户的 `owner` 必须是当前的 `authority`（通常是签名者或 PDA）。

- Sol-13: 金库余额是否使用了风险较高的“严格等值”比较？
    - 描述：若逻辑中使用 `balance == constant` 判断状态，攻击者可以通过直接向合约账户“捐赠”微量代币，破坏余额等式，导致程序逻辑锁死。
    - 建议：使用 `>=` 大于等于号进行余额判断，或者在逻辑内部维护一套记账变量，不完全依赖链上实时余额。

- Sol-14: 链下签名的授权参数是否在链上进行了保底校验？
    - 描述：虽然有后端签名授权，但如果链上不校验价格、滑点、截止时间等参数，一旦后端被攻破或签名泄露，合约将毫无防线。
    - 建议：链上必须对所有关键业务参数设置合理的硬边界（如最大滑点限制）。

- Sol-15: 交互方资产扣除是否确认了合法授权？
    - 描述：在P2P业务中，如果没有确认资产扣除方已签名，攻击者可能利用程序权限擅自移动他人资产。
    - 建议：确保所有涉及代币转移的操作中，`from` 账户的权限拥有者必须是交易的签名者。

- Sol-16: 资产锁定后的生命周期是否形成了闭环？
    - 描述：充入合约的资产如果没有明确的提取、销毁或在极端情况下的紧急退出路径，资产将永久锁死。
    - 建议：为每一种资产流入路径设计对应的流出逻辑，并进行压力测试确保退出路径畅通。

- Sol-17: 是否适配了通缩代币 (Fee-on-Transfer)？
    - 描述：某些代币在转账时会扣除手续费。如果程序假设“转出多少，到账多少”，会导致内部记账多于实际库存，最终引发提现溢出。
    - 建议：在转账前后分别查询余额，以 `balance_after - balance_before` 作为实际到账金额进行入账记录。

- Sol-18: 是否考虑了非标准 SPL 代币或 Token2022 扩展的兼容性？
    - 描述：Token2022 支持 Transfer Hook、永久委托等新特性，若程序未针对性处理，可能导致逻辑被 Hook 恶意劫持。
    - 建议：明确程序支持的代币标准，对不支持的扩展字段在入账时进行拦截或特殊处理。

- Sol-19: 极小金额的“粉尘”资金是否会造成逻辑锁死？
    - 描述：因精度截断产生的微量余渣（Dust）可能导致 `account.amount == 0` 的检查失败，从而无法注销账户。
    - 建议：在清理账户时，对极小金额（如小于 100 lamports 或最小代币单位）进行忽略或统一归入国库处理。

- Sol-20: 是否使用了安全的资产转移函数 (Safe Transfer)？
    - 描述：原生转账可能因目标地址不支持（如缺少 ATA）而失败，导致整个事务回滚。
    - 建议：使用 Anchor 的 `token::transfer` 并确保在转账前已为接收者创建了关联代币账户（ATA）。

## 三、 业务逻辑与状态管理 

- Sol-21: 紧急Pause机制是否覆盖了所有资金入口？
    - 描述：当发现漏洞时，如果只暂停了部分功能，攻击者仍可通过未封堵的入口提取资金。
    - 建议：实现全局开关，并在所有涉及 `transfer`、`mint`、`burn` 的指令顶部添加 `is_paused` 检查。

- Sol-22: 业务逻辑中的时间参数是否绝对有效？
    - 描述：如果仅检查 `start < end` 而不校验 `start > now`，攻击者可能利用过去的时间戳绕过某些限制。
    - 建议：确保时间戳逻辑符合业务预期，并考虑网络延迟对 `Clock::get()?.unix_timestamp` 的影响。

- Sol-23: 核心数学模型的语义边界是否在入库前校验？
    - 描述：例如 AMM 曲线中，如果参数输入导致分母趋近于 0，可能产生极端数据。
    - 建议：对入库的核心参数（如费率、杠杆倍数）设置 `MAX/MIN` 限制，防止数据极端化导致的系统性风险。

- Sol-24: 阶梯配置或费率表是否保持了逻辑连贯性？
    - 描述：如果不同阶梯的费率设置错误（例如高等级费率反而高于低等级），会造成业务逻辑死区。
    - 建议：在设置配置时增加单调性（递增/递减）校验。

- Sol-25: 用户退出后，账户中的状态数据是否彻底清理？
    - 描述：如果用户提现后未重置对应的 `amount` 或 `flag`，可能导致其再次调用时产生“双花”或逻辑重入。
    - 建议：遵循“提现即清零”原则，确保所有关联状态在业务结束时被重置。

- Sol-26: 关键角色地址是否强制了非零校验？
    - 描述：将管理员或收费地址误设置为 `Pubkey::default()`（全零地址）会导致权限丢失或资产永久消失。
    - 建议：在所有 `SetAddress` 函数中增加对目标地址是否为默认地址的校验。

- Sol-27: 是否清理了冗余的死代码与失效校验？
    - 描述：永不触发的 `if` 分支或过时的校验逻辑可能隐藏逻辑矛盾，给审计和后期维护带来隐患。
    - 建议：定期重构代码，删除未使用的变量和指令。

- Sol-28: 基于元数据的资产鉴伪是否溯源至发行根源？
    - 描述：如果根据 NFT 的名称或符号判断其价值，攻击者可以铸造同名的伪造 NFT。
    - 建议：必须校验资产的 `mint` 地址、`Creator` 地址或 `Collection` 属性，确保来源唯一且可信。

- Sol-29: 双向对称业务逻辑是否保持一致？
    - 描述：存入和取出、买入和卖出的逻辑在处理数据重置（如奖励累积器）时必须完全对称，否则会产生漏钱点。
    - 建议：将对称逻辑抽象为同一底层函数处理，减少手动编写两套逻辑产生的偏差。

- Sol-30: 状态隔离是否防止了通过备用接口绕过暂停？
    - 描述：攻击者可能通过一些辅助性接口（如管理员迁移接口）在系统暂停期间移动资产。
    - 建议：确保“暂停”状态是全程序层面的，任何影响状态变更的入口都应受到约束。

## 四、 算术运算与精度控制 

- Sol-31: 所有的算术操作是否都防范了静默溢出？
    - 描述：在 Rust/Solana 中，不加保护的加减法在溢出时会触发 panic 或在某些版本下回绕，导致计算错误。
    - 3建议：强制使用 `checked_add`、`checked_sub`、`checked_mul`、`checked_div`，并妥善处理 `None` 返回值。

- Sol-32: 计算顺序是否严格遵守“先乘后除”原则？
    - 描述：先执行除法会导致精度损失被放大，尤其是在处理代币单位时，可能导致金额大幅缩水。
    - 建议：始终先进行乘法运算，并在必要时使用 `u128` 承接中间结果以防止中间过程溢出。

- Sol-33: 向协议支付的金额（如手续费）是否使用了向上取整 (Ceiling)？
    - 描述：如果手续费向下取整，攻击者可以通过极小额交易让手续费归零，从而“吸血”协议。
    - 建议：对于应支付给系统的金额，使用 `(a * b + (d - 1)) / d` 的方式实现向上取整。

- Sol-34: 协议发放的奖励（如利息）是否使用了向下取整 (Floor)？
    - 描述：如果奖励向上取整，长期运行会导致协议国库发放的资金略多于账面资金，造成空库。
    - 建议：对于系统支出的金额，默认使用向下取整，确保国库资产支出始终保守。

- Sol-35: 是否存在低精度导致的薅羊毛漏洞？
    - 描述：当计算公式为 `amount * fee / scale` 时，若 `amount * fee < scale`，结果为 0。攻击者可借此免除手续费。
    - 建议：在手续费逻辑中增加 `require!(fee_result > 0)` 校验，或设定最小起征额。

- Sol-36: 复利与指数运算是否考虑了时间片切割导致的精度累积偏差？
    - 描述：复杂的利息计算在频繁操作下，取整误差会累加，可能导致系统负债。
    - 建议：使用高精度数学库（如 `FixedPoint`）或增加虚拟头寸记录来对齐误差。

- Sol-37: 算术结果归零是否会导致逻辑DoS？
    - 描述：某些逻辑中如果 `amount / price` 结果为 0 且后续有 `require!(result > 0)`，用户可能因资金量太小而无法提取。
    - 建议：在设计逻辑时确保即便是极小金额也能正常结清退出，或提供强制清理接口。

- Sol-38: 代币兑换撮合的舍入规律是否具有一致性？
    - 描述：如果 A 给 B 钱是向下取整，而 B 收到钱是按向上取整记账，系统会平空多出或少掉代币。
    - 建议：确保在单次事务中，资金流出的减少量必须大于等于资金流入的增加量。

- Sol-39: 是否存在由于底层数据类型强转导致的溢出Type Overflow？
    - 描述：将 `u64` 强转为 `u32` 时，高位会被截断，导致数值突变。
    - 建议：使用 `try_from` 或 `as` 转换前进行边界检查，确保数值在目标类型的表示范围内。

- Sol-40: 复杂计算公式是否进行了单调性验证？
    - 描述：不合理的数学模型可能导致“输入越多收益越少”的异常情况，攻击者可以找到局部最优解进行套利。
    - 建议：对核心算法进行数学建模，确保收益随投入量单调非递减。

## 五、 外部依赖与组合性 

- Sol-41: 预言机 Sequencer 活性校验是否完备？
    - 描述：在 L2 或跨链环境中使用预言机（如 Chainlink）时，如果 Sequencer 宕机且不校验，价格可能停留旧值。
    - 建议：在获取价格前，先检查预言机提供的 Sequencer 状态标志。

- Sol-42: 程序设计是否隔离了硬编码地址风险？
    - 描述：硬编码 Program ID 或 Mint 地址会导致代码难以在 Devnet/Testnet 或其他克隆链上部署。
    - 建议：使用配置账户注入外部程序地址，并在初始化时设定，增强程序的灵活性和可测性。

- Sol-43: 跨组件拦截规则是否会导致协议 DoS？
    - 描述：如果依赖的代币带有黑名单（如 USDC），一旦某用户被黑，其在协议中的资金可能锁定，甚至拖累整个池子。
    - 建议：考虑资产被外部冻结时的隔离方案，确保单个用户的资产风险不传染给协议全局。

- Sol-44: 外部依赖合约是否存在升级或接口变动风险？
    - 描述：依赖的外部协议如果是非不可篡改的，其升级可能改变返回值语义，导致你的合约逻辑崩溃。
    - 建议：尽量依赖经过时间考验且权限治理透明的协议，并对外部返回值进行防御性编程。

- Sol-45: 多步操作中的事务原子性是否得到保障？
    - 描述：在调用外部合约时，如果逻辑依赖外部执行成功而未对失败进行回滚，可能导致本合约状态异常。
    - 建议：利用 Solana 事务的原子性特点，确保跨合约调用失败时整个交易回滚。

## 六、 可观测性与代码规范 

- Sol-46: 关键治理状态变动是否抛出了Events？
    - 描述：没有日志的参数修改会让离线监控（如后端解析器）无法追踪协议变动。
    - 建议：对所有管理员操作、权限转移、核心参数修改使用 `emit!` 记录详细日志。

- Sol-47: 复用的数据结构是否带有事件区分标识TypeID？
    - 描述：如果多个操作抛出结构相同的事件，后端将难以区分是“存款”还是“取款”。
    - 建议：在事件结构体中增加一个 `action_type` 枚举字段。

- Sol-48: 是否使用了强类型的状态引用与架构解耦？
    - 描述：直接操作原始账户数据（Raw Data）极易出错且难以阅读。
    - 建议：充分利用 Anchor 的 `Account<'info, T>` 宏，利用框架层的类型检查减少手动逻辑。

- Sol-49: 账户空间申请Space是否进行了优化并考虑了冗余？
    - 描述：Solana 账户租金昂贵。空间申请不足会导致无法存储更多数据，申请过多则浪费用户租金。
    - 建议：精确计算需要的字节数（加上 8 字节的 Anchor 判别符），并为未来扩展预留适量空间。

- Sol-50: 代码逻辑是否与文档说明保持绝对一致？
    - 描述：文档宣称有 5% 手续费而代码写了 10%，或宣称有锁定执行而代码中缺失，属于“文档误导型”安全风险。
    - 建议：在审计前进行一致性核对，确保白皮书、代码注释与实际执行路径相符。
